(SQL10)数据库复习

Destiny2025-12-062026-01-17

1.绪论

数据库的四个基本概念

  • 数据（data）：描述事物的符号，数据库中存储的基本对象
  • 数据库（DataBase，DB）：长期存储在计算机内，有组织的，可共享的大量数据的集合（DB 的三个特点）
  • 数据库管理系统（DataBase Management System，DBMS）：位于用户和计算机系统之间的一层数据管理软件
  • 数据库系统（DataBase System，DBS）：由数据库，数据库管理系统，应用程序，数据库管理员（DB）组成

数据库系统的特点

  • 数据结构化
  • 数据共享性高，冗余度低，且容易扩充
  • 数据独立性高
    • 物理独立性：应用程序与物理存储相互独立，数据的物理存储改变，应用程序不改变
    • 逻辑独立性：应用程序与逻辑结构相互独立，数据的逻辑结构改变，应用程序不改变
  • 数据由数据库管理系统统一控制和管理

数据模型

数据模型包括概念模型（信息模型）和逻辑模型 / 物理模型

  • 概念模型：按照用户的观点建模，用于数据库设计，表示方法有 E-R 模型
  • 逻辑模型：按照计算机的观点建模，用于 DBMS 实现，包括网状模型，层次模型，关系模型（重点）
  • 物理模型：计算机最底层对数据的抽象

数据模型的组成要素

数据模型的组成要素：数据结构，数据操作，数据的完整性约束条件

  • 数据结构：描述系统的静态特性（描述数据库的组成对象以及对象之间的联系）
  • 数据操作：描述系统的动态特性（对数据库中各种对象的实例所允许的操作的集合，类型包括查询，更新，删改）
  • 数据的完整性约束条件：用以限定符合数据模型的数据库的状态以及状态的变化，以保证能够数据的正确，有效与相容，完整性约束条件是一组完整性规则的集合

数据库系统的模式结构

数据库系统采用三级模式结构：外模式，模式（逻辑模式），内模式

  • 外模式 / 子模式 / 用户模式：数据库用户能够看见和使用的局部数据的逻辑结构和特征的描述，数据库用户的数据视图，时雨某应用有关的数据的逻辑表示
  • 模式 / 逻辑模式：数据库中全体数据的逻辑结构和特征的描述，是所有用户的公共数据视图
  • 内模式 / 存储模式：数据物理结构和存储方法的描述，是数据在数据库内部的组织方式

4.数据库安全

数据库安全

数据库安全指的是保护数据库以防止不合法的使用造成数据泄露，更改和破坏

数据库安全包括两部分，数据的安全性控制和数据的完整性控制

数据安全性控制

数据安全性控制包含以下几个策略：

（1）用户标识与认证：最外层安全保护措施

  • 标识：系统采用一定的方式识别其用户或应用程序的名字或身份
  • 认证：系统在用户或应用程序登录时判断其是否为合法的授权用户

（2）存取控制

  • 存取控制包括用户权限定义和合法权限检查两部分，二者共同构成 DBMS 的安全子系统
  • 存取控制可以分类为自主存取控制（DAC）和强制存取控制（MAC），其中前者又称为基于角色的存取控制
  • SQL 可以授予用户两种权限：用户级权限和关系级权限

（3）视图机制

（4）审计

（5）数据加密

数据库系统的映象

数据库系统采用两级映像功能，能够保持数据独立性

  • 外模式 / 模式映像：保证数据的逻辑独立性
    • 当模式改变时，对外模式 / 模式映像做出改变，保证外模式不变，应用程序由外模式编写，从而应用程序不变
  • 模式 / 内模式映像：保证数据的物理独立性
    • 当数据库存储结构改变时，对模式 / 内模式映像做出改变，使模式不变，进而应用程序不变

7.数据库设计

数据库设计的步骤

数据设计分为以下六个步骤（注意顺序）：

  • 需求分析
    • 调查机构情况与熟悉业务活动，明确用户的需求，确定系统的便捷，生成数据字典和用户需求规格说明书
  • 概念结构设计
    • 将需求分析得到的用户需求抽象为概念模型，绘制 E-R 图
  • 逻辑结构设计
    • 把 E-R 图转换为与 DBMS 相符合的逻辑结构（包括数据库模式和外模式），例如 E-R 图向关系模型转换
    • 根据规范化理论对关系模型进行优化
    • 设计用户子模式
  • 物理结构设计
    • 关系模式选择存取方法
    • 设计关系、索引等数据库文件的物理存储结构
  • 数据库实施
    • 建立实际运行的数据库结构
    • 试运行
    • 装入数据
  • 数据库运行和维护
    • 维护数据的安全性、完整性控制以及系统的转储和恢复
    • 性能的监督、分析和改进
    • 增加新功能
    • 发现错误和修改错误

10.数据库恢复

事务的概念

  • 事务是数据库操作序列，这些操作要么全做，要么不做，是不可分割的工作单位
  • 一个事务可以是一条 sql 语句，一组 sql 语句，或者整个程序，一般而言，一个程序包括多个事务
  • 事务是恢复的基本单位，也是并发控制的基本单位

定义事务的 SQL 语句

  • 开始语句：begin transaction;
  • 正常结束语句：commit;表示提交事务的所有操作
  • 异常结束语句：rollback;表示回滚，即事务运行中发生了某种故障，需要对已完成的操作进行撤销，回滚到事务的开始状态

事务的特性

事务具有以下四个特性（ACID），简称酸性：

  • 原子性：一个事务要么全做，要么全不做
  • 一致性：事务执行的结果必须一致，如果一个事务被迫中断，即事务中的一些操作已经完成，其余操作还未完成，已完成的操作对数据库造成的修改如果已经存入物理数据库，就会造成数据库不一致的状态，因此事务要么全做，要么全不做，一致性和原子性是密切相关的
  • 隔离性：一个事务的执行不能被其他事务所干扰
  • 持续性：也称为永久性，一个事务一旦提交，它对数据库中数据的改变应该是永久性的

数据库故障

数据库故障分为四类：

  • 事务内部故障
    • 事务内部的故障有的可以通过事务程序本身发现，有的是非预期的、不能由事务程序处理的
    • 事务故障的情况下，数据库可能处于不正确的状态，恢复程序要在不影响其他事务运行的情况下，强行回滚该事务，即撤销该事务所造成的修改，这类恢复操作称为事务撤销（UNDO）
  • 系统故障（软故障）
    • 系统故障是指造成系统停止运转的任何事件，使得系统要重新启动（例如硬件错误，操作系统故障，DBMS 代码错误，系统断电）
    • 系统故障的情况下，在系统重新启动后，回复子系统除了需要撤销未完成的事务以外，还需要重做（REDO）所有已提交的事务
  • 介质故障（硬故障）
    • 外存故障，如硬盘损坏，刺头碰撞，瞬时强磁场干扰，发生频率最低，破坏性最大
  • 计算机病毒
    • 航子不考

11.并发控制

并发控制的概念

并发控制是指为了保证事务的隔离性和一致性，数据库管理系统需要对并发操作进行正确调度

并发控制的技术有：封锁（考试只考这一个），时间戳，乐观控制法，多版本并发控制

并发操作的影响

并发操作的主要影响是造成数据的不一致性：

  • 丢失修改：A 先读数据，B 再读数据，A B 都对旧数据做修改，导致一张票卖两个人的情况
  • 不可重复读：A 读完数据，B 对数据修改，A 想再读一次和之前做验算，发现两次读取结果不一样
  • 读脏数据：A 修改完数据，B 再读数据，因为某些原因撤销了 A，数据回到就值，导致 B 读到的数据和数据库中数据不一致

并发控制的基本方法

航子只考封锁这一种控制方法，锁的类型有两种：

  • 排它锁 / 写锁 / X锁
    • 若事务 T 对数据 A 上 X 锁，则 T 可以读取和修改 A，在 T 释放对 A 的 X 锁之前，其他事务不能再对 A 上任何类型的锁，从而保证其他事务对 A 既不能读取也不能修改
  • 共享锁 / 读锁 / S锁
    • 若事务 T 对数据 A 上 S 所，则 T 只能读取 A，但不能修改 A，在 T 释放对 A 的 S 锁之前，其他事务只能对 A 上 S 锁，不能上 X 锁，从而保证其他事务可以对 A 读取，但不能对 A 修改

封锁类型的相容矩阵：有 S 后只能加 S 不能加 X，有 X 以后 S 和 X 都不能加

封锁协议

封锁协议用于约定何时申请 S 锁 / X 锁，锁的持续时间，何时释放锁，共有三级封锁协议：

  • 一级封锁协议：事务 T 在修改数据 R 之前必须对其加 X 锁，直到事务结束才能释放
  • 二级封锁协议：在一级封锁协议的基础上，事务 T 在读取数据 R 之前必须对其加 S 锁，直到读取结束才能释放
  • 三级封锁协议：在一级封锁协议的基础上，事务 T 在读取数据 R 之前必须对其加 S 锁，直到整个事务结束才能释放

死锁和活锁

活锁指的是某个事务长期无法访问数据，死锁指的是两个 / 多个事务互相持有对方对数据的锁，导致谁都无法执行

  • 活锁的解决办法：先来先服务
  • 死锁的解决办法：分为①预防②诊断与解除
  • 死锁的预防：
    • 一次封锁法：每个事务必须一次将所有要使用的数据全部枷锁，否则就不能继续执行
    • 顺序封锁法：预先对数据对象规定一个封锁顺序，所有事务按这个顺序实施封锁
  • 死锁的诊断与解除：
    • 超时法：如果一个事务的等待时间超过了规定的时限，就认为发生了死锁
    • 等待图法：构建事务等待图（有向图：节点表示事务，有向边表示等待关系，T1→T2T_1\to T_2T1​→T2​ 表示 T1T_1T1​ 等待 T2T_2T2​ 的锁），等待图存在回路即有死锁
      • 死锁解除：选择一个处理死锁代价最小的事务，将其撤销（UNDO），释放该事务持有的所有的锁

可串行化调度

可串行化调度：多个事务并发执行的结果，等价于这些事务按某种次序串行执行的结果

一个给定的并发调度是正确的，当且仅当它是可串行化的，比如给定事务 T1T_1T1​ 和 T2T_2T2​，若 T1T_1T1​ 和 T2T_2T2​ 的并发执行结果与 T1,T2T_1,T_2T1​,T2​ 或 T2,T1T_2,T_1T2​,T1​ 任意一个执行顺序相同，就认为该并发调度是正确的

冲突可串行化调度

冲突操作：不同事务对同一数据的读写操作，写写操作

不同事务的冲突操作和同一事务的两个操作是不能交换的，如果发生交换会产生错误（也就是调度序列中，冲突操作之间的相对位置不能被调换）

基于此，如果调度能够保证冲突操作的相对次序不变的前提下做可串行化，则称其为冲突可串行化的调度

冲突可串行化的调度一定是可串行化调度，但可串行化调度不一定是冲突可串行化调度（比如连续的全写调度序列）

两段锁协议

两段锁协议又称为 2PL，要求所有事务必须分两个阶段对数据项加锁和解锁，即事务 TTT 的所有对锁的操作可以严格分为两段，前半段全是加锁，后半段全是解锁，一旦开始解锁就只能执行解锁，不能再执行加锁，这两段也被称为：

  • 获得封锁（扩展阶段）：再对任何数据进行读、写操作之前，事务首先要活的对该数据的封锁
  • 释放封锁（收缩阶段）：在释放一个锁之后，事务不在申请和获得其他任何封锁

若并发执行的所有事务都遵循两段锁协议，则这些事务的任何并发调度策略都是可串行化的
