地址空间

在内存中，从高地址到低地址，自上而下依次是：

  • 静态数据区（全局和静态变量）
  • 代码区
  • 程序运行栈
  • 动态内存区（内存堆）

程序运行栈设计：

  • 子程序 / 函数运行时所需要的基本空间：活动记录
  • 进入子程序 / 函数时分配，地址空间向下生长（从高地址到低地址）
  • 从子程序 / 函数返回时，当前运行栈被废弃
  • 递归调用的同一个子程序 / 函数，每次调用都会获得独立的运行栈空间，从而保证了递归程序和多线程程序的正确运行

寄存器的分配

（1）全局寄存器分配：

  • 全局寄存器分配：这里我们所说的全局是相对于基本块而言，而不是程序全局而言，全局寄存器的分配对象主要是函数的局部变量，包括函数入口参数
  • 分配原则：优先分配给跨基本块仍然活跃的变量，尤其是循环体内最活跃的变量；局部变量参与全局寄存器分配，但全局变量和静态变量一般不参与全局寄存器分配（这是因为寄存器属于专属线程，我们通常是基于线程安全的考虑）

全局变量和静态变量一般不参与全局寄存器分配，及时他们在某个循环体中被多次访问：这是因为当一个每个线程都有自己专属的全局寄存器，全局 / 静态变量在不同的线程之间是由不同的全局寄存器来存储的，这就会导致不同线程之间全局 / 静态变量的值不一致的问题

比如下面这个例子，如果把 s_cs\_cs_c 用全局寄存器来存储，Thread1 切换到 Thread2 时，会换一个全局寄存器来存 s_cs\_cs_c 的值，而 Thread2 中的全局寄存器此前被初始化为 0，第一次调用得到的 s_c=1s\_c=1s_c=1 的值就会丢失，又会从 0 开始加载

常用全局寄存器分配方法

（1）引用计数法

通过统计变量在函数内被引用的次数（这里的引用包括定义和使用，比如 a=a+1a=a+1a=a+1 视作 a 被引用了两次），并根据被引用的特点赋予不同的权重（在循环体中被引用一次实际代码运行时可能被引用了多次，可以适当增加权重），最终为每个变量计算出一个唯一的权值，根据权值的大小排序，将全局寄存器依次分配给权值最大的变量

分配算法：如果有 N 个全局寄存器可供分配，则权重前 N 大的变量拥有全局寄存器，其余变量在程序运行栈（活动记录）分配存储单元

问题：不再使用的变量不能及时释放寄存器（比如某个变量在前期大量使用，后期不再使用，但是其权重整体很高，则可能在后期也一直占用全局寄存器）

（2）着色图算法

通过构建变量之间的冲突图，在图上应用着色算法，将不同的全局寄存器分配给有冲突的变量

  • 首先通过数据流分析，构建变量的冲突图（冲突图的节点是待分配全局寄存器的变量，当两个变量中的一个变量在另一个变量定义 / 赋值处是活跃的，他们之间就有一条边的连接，而所谓变量 i 在代码 n 处活跃，指的是程序运行时变量 i 在 n 处拥有的值，在从 n 出发的某条路径上会被使用）
  • 如果可以分配 k 个全局寄存器，我们就尝试使用 k 种颜色给冲突图着色
  • 我们在冲突图中找到第一个连接边数目严格小于 K 的节点，把这个节点从图 G 中移除，得到 G’
  • 重复上述步骤，直到再也无法从 G‘ 中移除任何节点（G 变为平凡图或不存在连边数目小于 K 的节点）
  • 如果某一时刻，图上所有节点连接的边数都 >= K，此时我们就选择合适的节点，直接把它移走，同时标记这个节点不能分配全局寄存器，这是为了我们的图最终能化到最简（只有一个节点）
  • 最后给图上剩余的最后一个节点选取一种颜色，然后按照节点被移走的顺序，反向把节点和边添加进去，并依次给新加入的节点选取颜色，保证有链接边的节点着不同的颜色

临时寄存器分配方法：

  • 进入基本块时，清空临时寄存器池
  • 为当前中间代码生成目标代码时，无论临时变量还是局部变量（甚至是全局变量和静态变量），如果需要使用临时寄存器，都可以向临时寄存器池申请
  • 临时寄存器池接到申请以后，如果寄存器池中有空闲寄存器，则直接把该空闲寄存器拿去使用
  • 如果寄存器池中没有空闲寄存器，则选取一个在即将生成的代码中不会被使用的寄存器，把这个寄存器中原本存储的值写回相应的内存空间（一般是栈区），标记该寄存器被新的变量占用，把该寄存器返回
  • 在基本块结尾，或函数调用发生前，将寄存器池中所有被占用的临时寄存器写回相应的内存空间，清空临时寄存器池
