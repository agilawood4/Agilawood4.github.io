数据库保护

数据库系统的安全性

数据库的安全性是指保护数据库以防止不合法的使用所造成的数据泄露，更改和破坏，其包括两个方面的含义：

  • 向授权用户提供可靠的信息服务
  • 拒绝对数据的非授权存取访问请求，保证数据的可用性，完整性和一致性，进而保护数据库所有者和使用者的合法权益

数据库的安全性控制

包含数据库系统的计算机系统安全模型：

用户标识与鉴别

用户认证和标识是系统提供的最外层安全保护措施：

  • 标识是指系统采用一定的方式标识其用户或应用程序的名字或身份（知道用户 / 程序是谁）
  • 认证是指系统在用户或应用程序登录时刻判断其是否为合法的授权用户（判断用户 / 程序有没有权限）

常用的标识和鉴别方法是用户名和口令

存取控制

存取控制确保合法用户按照指定的权限使用 DBMS 和访问数据，而非法用户或不具有相关权限的用户则不能

存取控制机制主要包含两个部分：

  • 用户权限定义：把用户权限记录到数据字典中，形成安全规则或授权规则
  • 合法权限定义：每当用户发出数据库操作请求时，DBMS 根据数据字典中的安全规则进行合法权限检查，决定是否接受用户的操作请求

用户权限定义和合法权限检查机制一起组成了 DBMS 的安全子系统

存取控制方法分类

自主存取控制（DAC，discretionnary access control）：用户对不同的数据对象有不同的存取权限，不同的用户对同一对象也有不同的权限，同时，用户可以把其拥有的权限转授给其他用户

强制存取控制（MAC，mandatory access control）：每一个数据对象被标以一定的密级，每一个用户也被授予某一个级别的许可证，对于任何一个对象，只有满足合法许可证的用户才可以存取

自主存取控制

根据预先定义的用户权限进行存取控制，用户权限是指用户对数据对象允许执行的操作类型，由数据对象和操作类型两个要素组成

对于用户存取权限的定义称为授权，在授权中应该指明：用户名，数据对象名，允许的操作类型

强制存取控制

在 MAC 中，DBMS 所管理的全部实体被分为主体和客体两部分：

  • 主体：系统中的活动实体，既包括 DBMS 所管理的实际用户，也包括代表用户的各进程
  • 客体：系统中的被动实体，是受主体操纵的，包括文件，基本表，索引，视图等

对于主体和客体，DBMS 为他们的每个实例制定一个敏感度标记（Label），敏感度标记分为若干个级别，如：绝密，机密，秘密，公开等，主体的敏感度标记称为许可证级别，客体的敏感度标记称为密级

MAC 机制通过对比主体的 Label 和客体的 Label，最终确定主体能否存取客体：

  • 仅当主体的许可证级别大于或等于客体的密级时，主体才能够读取相应的客体
  • 仅当主体的许可证级别等于客体的密级时，主体才能写相应的客体（这里强调的必须是严格等于，大于也不行，因为高许可证级别的的主体如果能写低密级的客体，那么这个客体中就会有存入高密级的数据，可能会被其他低许可证级别的主体可见，从而造成数据泄露）

角色权限控制

角色可以是一组相关权限的集合，将角色授予用户可以实现权限分配，用户访问时根据所拥有的角色权限进行存取控制——RBAC，使用角色控制用户对数据对象的权限，可以大幅度简化权限的管理

比如我们可以设置学生权限 A 和教师权限 B，给每个学生统一赋予 A 权限，给每个教师统一赋予 B 权限，这样就不用每加入一个成员，就为其设计一次权限

SQL 数据安全性控制

在 SQL 中可以授予用户两类权限：

  • 用户级权限：数据库管理员为每个用户授予的特定权限，是对用户使用整个数据库权限的设定，与整个数据库相关，与数据库中具体的关系无关（能访问哪些数据库）
  • 关系级权限：是数据库管理员或数据库对象的拥有者为用户授予的与关系或视图有关的权限，这种权限是对用户使用关系和视图权限的限定（在数据库里能对哪些数据具体做什么操作）

数据安全性控制的其他方法

视图机制

为不同的用户定义不同的视图，可以将用户对数据的访问限制在一定的范围内，比如限制王平只能检索 Student 表中计算机系学生的学号和姓名：

审计

把用户对数据库的所有操作都自动记录下来放入审计日志中，DBA 可以利用审计跟踪的信息，重现导致数据库现有状况的一系列事件，找出非法存取数据的人，时间和内容等

数据加密

防止数据库中数据在存储和传输中失密，加密的基本思想是根据一定的算法将原始数据（明文）转化为不可识别的格式（密文），从而使得不知道解密算法的人无法获知数据的内容

数据完整性控制

数据完整性控制

数据完整性指的是数据的正确性和相容性：

  • 正确性：数据应具有合法的类型，并且在有效的取值范围之内
  • 相容性：表示同一个事实的两个数据必须相同，也就是不同表之间的信息不能起冲突

数据库能否保持完整性关系到数据库能否能够真实的反应现实世界：

  • 数据完整性控制是为了防止数据库中存在不符合语义的数据，防止错误信息的输入和输出
  • 数据安全性控制是保护数据库防止恶意的破坏和非法存取
  • 数据安全性防范的是非法用户和非法操作，数据完整性防范的是不合语义的数据

完整性约束条件

施加在数据库之上的语义约束条件称为数据库完整性约束条件，数据库系统依据完整性约束条件进行完整性检查，完整性约束条件的作用对象可以是列，元组，关系三种：

  • 列的约束：列的类型，取值范围，精度等
  • 元组的约束：元组中各个字段之间联系的约束
  • 关系的约束：若干元组之间，关系之间的联系的约束

关系模型中的三种完整性的约束条件：实体完整性，参照完整性，用户自定义完整性

完整性约束条件分类

完整性约束可以分为静态约束和动态约束：

  • 静态约束：数据库在每一确定状态数据对象所应满足的约束条件，它是反应数据库状态合理性的约束
  • 动态约束：数据库从一种状态转变为另一种状态时，新、旧值之间所应满足的约束条件，它是反应数据库状态变迁的约束

静态约束

静态列级约束：对一个列的取值域的说明，包括对数据类型（数据类型，长度，单位，精度等），数据格式，取值范围，取值集合，空值等的约束

静态元组约束：组成一个元组的各个列之间的约束关系（比如在教职工工资表中，教授的工资必须不少于讲师的工资）

静态关系约束：一个关系的若干元组或者若干关系之间常常存在的各种联系或约束，包括实体完整性约束，参照完整性约束，函数依赖，统计约束等（比如在教师信息表和授课表中，授课表中的教师 id 必须存在于教师信息表中）

动态约束

动态列级约束：修改列定义或列值时应满足的约束条件

动态元组约束：修改元组值时元组中各个字段间需要满足的约束

动态关系约束：加载关系变化前后状态上的限制条件

完整性控制

数据库的完整性控制包含以下三部分内容：

  • 定义功能：提供定义完整性的约束条件的机制
  • 检查功能：检查用户发出的操作请求是否违背了完整性约束条件
  • 违约相应：如果违背了完整性约束条件，采取一定的措施来保证数据完整性

完整性约束条件按照完整性检查的时机分为立即执行约束和延迟执行约束：

  • 立即执行约束：在执行用户事务的过程中，在一条语句执行完后立刻进行完整性约束的检查，如果违反了完整性约束，系统会拒绝该操作
  • 延迟执行约束：在整个用户事务执行完毕以后，再进行完整性约束检查，检查正确才进行提交，否则系统拒绝整个事务

完整性规则的表示

一条完整性规则可以用一个五元组（D，O，A，C，P）来描述，其中：

  • D(Data)：约束所作用的数据对象
  • O(Operation)：触发完整性检查的数据库操作
  • A(Assertion)：数据对象必须满足的断言或语义约束
  • C(Condition)：选择 A 作用的数据对象值的谓词
  • P(Procedure)：违反完整性规则时所触发的过程
