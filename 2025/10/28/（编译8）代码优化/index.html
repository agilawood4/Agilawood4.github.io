代码优化

代码优化：指编译程序为了生成高质量的目标程序而做的各种加工和处理

目的：提高目标代码运行效率（包括空间效率和时间效率）

原则：进行优化必须严格遵循 “不能改变原有程序语义” 的原则

记住这三个的顺序，从左到右依次是：局部优化、循环优化、全局优化，这个顺序表名，只要做一些简单的处理，就能够得到明显的优化效果，而如果要进一步提高优化效果，则需要逐步付出更大的代价

优化的分类（从优化层次来看，分为和机器是否有关）：

  • 独立于机器的优化：即和目标机无关的优化，通常是在中间代码上进行的优化（和机器无关，只针对代码，在任何机器都成立）
  • 与机器有关的优化：充分利用系统资源（指令系统，寄存器资源）

优化的分类（从优化涉及的范围来看）：

  • 局部优化：在基本块内进行的优化
  • 循环优化：对循环语句所生成的中间代码序列上所进行的优化
  • 全局优化：跨越多个基本块的全局范围内的优化，即在非线性程序段上（包括多个基本块，GOTO，循环）的优化，需要进行全局控制流和数据流分析，比较复杂

基本块

基本块定义：

  • 基本块中的代码是连续的语句序列
  • 程序的执行（控制流）只能从基本块的第一条语句进入
  • 程序的执行只能从基本块的最后一条语句离开

流图的建立

流图是一种有向图，其节点为基本块，如果在某个执行序列中，B2 的执行紧跟在 B1 之后，则从 B1 到 B2 有一条有向边，此时我们称 B1 为 B2 的前驱，B2 为 B1 的后继，有两种情况：

  • 从 B1 的最后一条语句有条件或无条件转移到 B2 的第一条语句
  • 按照程序的执行次序，B2 紧跟在 B1 之后，且 B1 没有无条件转移到其他基本块

有了基本块的概念以后，代码优化可以分为基本块内优化，基本块之间的优化两部分

基本块内优化

（1）利用代数性质（代数变换），简化表达式

  • 在编译的时候就完成常量表达式的计算，整数类型与实型的转换，例如：
  • 下标变量引用时，其地址分为基址和可变偏移两部分，在编译时可以预先算好基址部分，运行时只需要计算可变部分即可
  • 运算强度削弱：用一种需要较少执行时间的运算代替另一种运算，以减少运行时的运算强度（时空开销），利用机器硬件所提供的一些功能，如左移，右移操作做乘法或除法，具有更高的代码效率
  • 复写（copy）传播：如x:=y这样的语句称为复写语句，由于 x 和 y 取值相同，当满足一定条件时，在该复制语句下面出现的 x 可以用 y 来替代，如果是x:=3则这种形式，则称为常量传播
  • 删除冗余代码：冗余代码就是毫无实际意义的代码，又称为死代码（dead code）或无用代码（useless code）

（2）消除公共子表达式

DAG 图：有向无环图，用来表示基本块内各个中间代码之间的关系，可以使用 DAG 图消除公共子表达式

基本块的 DAG 图表示：

  • 图的叶节点由变量名或常量所标记（对于基本块内先引用再赋值的变量，可以采用变量名+下标 0 的方式，命名其初值）
  • 图的中间节点由中间代码的操作符所标记，代表着基本块中一条或多条中间代码
  • 基本块中的变量的最终计算结果都对应着图中的一个节点，具有初值的变量，其初值和终值可以对应不同的节点

借助 DAG 设计消除局部公共子表达式的算法：本质上应该分成两部分，先由初始的中间代码生成 DAG 图，再从 DAG 图导出化简后的中间代码

DAG 图的生成

（1）生成 DAG 图的算法

输入：基本块内的中间代码序列

输出：完成局部公共子表达式删除后的 DAG 图

方法：

  • 首先建立节点表，该表记录了变量名和常量值，以及他们当前所对应的 DAG 图中节点的序号，该表初始状态为空
  • 从第一条中间代码开始，按照以下规则建立 DAG 图
  • 对于形如z = x op y的中间代码，其中 z 为记录计算结果的变量名，x 为左操作数，y 为右操作数，op 为操作符：首先在节点表中寻找 x，如果找到，则记录下 x 当前对应的节点号 i；如果未找到，在 DAG 图中新建一个叶节点，假设其节点号仍为 i，标记为 x（如果 x 是变量名，则标记为 x0），在节点表中增加新的一项（x, i），表明二者之间的对应关系，右操作数 y 与 x 同理，假设其对应节点号为 j
  • 在 DAG 图中寻找中间节点，其标记为 op，且左操作数节点号为 i，右操作数节点号为 j，如果找到，记录下其节点号 k，如果未找到，在 DAG 图中新建一个中间节点，假设其节点号仍为 k，并将节点 i 和 j 分别与 k 相连，作为其左子节点和右子节点
  • 在节点表中寻找 z，如果找到，则将 z 所对应的节点号改为 k，如果未找到，在节点表中新建一项（z, k），表明二者之间的对应关系

对函数，数组，指针不进行优化，做保守处理，形如x=a[i], a[j]=y, z=a[i]，如果 i == j 则此时 x 和 z 的值不相同，我们无法简单确定能否对 x 和 z 做代码优化，所以我们一律不做优化

节点表的建立

例1：我们需要为所有局部变量（程序中原本就有的变量）和临时变量（我们为保存中间运算结果额外设的变量）设计一张表，不重不漏地存他们的变量名和节点次序（单调递增的编号）

  • 记录局部变量 c，编号为 1
  • 记录临时变量 t1，编号为 2（这里 c 和 -c 因为值不同，看成两个节点，-c 需要一个额外的临时变量 t1 来存储）
  • 记录局部变量 b，编号为 3
  • 记录临时变量 t2，编号为 4，存 b*t1 的结果
  • 记录临时变量 t3，注意，t3 和 t1 的值相同，此时不需要再开辟新的节点编号，而是让 t3 和 t1 共享同一个节点编号 2 号即可
  • 记录临时变量 t4，注意，t4 和 t2 的值相同，也不需要重新开辟，让 t4 和 t2 共享一个节点编号 4 号即可
  • 记录临时变量 t5，编号为 5，存 t2+t4 的结果
  • 记录局部变量 a，注意，a 和 t5 的值相同，也不需要重新开辟，让 a 和 t5 共享一个节点编号 5 号即可

例2：如果中间代码形如z=x，即出现局部变量被重新赋值的情况，只需要找到 x 所对应的节点 i，把 z 对应的节点编号改为 i 即可

  • 记录局部变量 c，编号为 1
  • 记录临时变量 t1，编号为 2，存 -c 的值
  • 记录局部变量，编号为 3
  • 记录临时变量 t2，编号为 4，存 b*t1 的结果
  • 此时局部变量 c 被重新赋值，我们找到它的节点（c, 1），此时 c 被赋值为 t1，t1 的编号为 2，因此我们更新 c 的编号为 2，c 的节点此时对应（c, 2）
  • 记录临时变量 t3，编号为 5，存 -c 的值，此时 c 要用新的节点，也就是（c, 2）中的值
  • 记录临时变量 t4，编号为 6，存 b*t3 的结果
  • 记录临时变量 t5，编号为 7，存 c+t4 的结果
  • 记录局部变量 a，a 和 t5 的值相同，二者共用同一个节点编号 7 即可

由 DAG 图导出中间代码

从 DAG 导出中间代码的启发式算法核心：变量的定义距离变量的使用处不能太远，我们希望临时变量产生后尽快被使用，而不是总占着寄存器的空间

输入：DAG 图

输出：化简后的中间代码序列

方法：

  • （1）初始化一个放置 DAG 图中间节点的队列（注意队列里只放中间节点，中间节点指的是 op 节点）
  • （2）如果 DAG 图中还有中间节点没有进入队列，则执行步骤 3，否则执行步骤 5
  • （3）选取一个尚未进入队列，但其所有父节点（这里其实是广义父节点，即所有祖先节点）都已经进入队列的中间节点 n，把 n 压入队列，或者选择一个没有父节点的中间节点，将其压入队列
  • （4）把 n 在第三步压入队列以后，接着进行检查，如果 n 的最左子节点符合步骤三的条件，则将其也压入队列，并且沿着当前节点的最左边，递归遍历最左子节点的最左子节点，把符合步骤三条件的这些中间节点依次压入队列，如果出现不符合步骤三条件的最左子节点，则执行步骤二
  • （5）把中间节点队列逆序输出（先队尾后队首），得到中间节点的计算顺序，将其整理为中间代码序列

全局优化

数据流分析

数据流分析：在控制流的基础上，增加数据在程序执行路径中如何流动的有关信息

例如：

  • 某个变量在某个特定的执行点（语句前后）是否还存存活
  • 某个变量的值，是在什么地方定义的
  • 某个变量在某一执行点上被定义的值，可能在其他哪些执行点被使用

数据流分析是全局优化的基础

程序的状态

程序的执行过程：程序状态的变换过程

程序状态由程序中的变量和其他数据结构组成，比如如果程序中总共只有三个变量 x y z，可以认为程序的状态由向量 (x,y,z)T(x,y,z)^T(x,y,z)T 组成：

  • 每一条执行指令都可能改变程序的状态（定义变量、注销变量、变量赋值）

通过数据流分析，可以了解程序的状态：

  • 如果得知在某条中间代码之后，无论程序在实际执行时通过哪条路径，某个变量都不会再被访问，那么该变量此前所保有的全局寄存器或临时寄存器就可以安全地被其他变量重新使用
  • 如果得知在程序的某个点上，对某个变量进行引用时，无论程序如何运行，该变量都仅有一个唯一的常量值，那么就可以将该常量值直接引入中间代码，在代码生成时得到更高效的指令

一种常用的数据流分析方法：达到定义

数据流分析方程

考察在程序的某个执行点的数据流信息：

out[S]=gen[S]∪(in[S]−kill[S])out[S]=gen[S]∪(in[S]-kill[S]) out[S]=gen[S]∪(in[S]−kill[S])

  • S 代表某条件语句，或某个语句集合，或某个基本快，或某个基本块集合
  • in[S]：表示进入 S 时的数据流信息
  • out[S]：表示在 S 末尾得到的数据流信息
  • gen[S]：表示 S 本身产生的数据流信息（比如在 S 中新定义的变量，或对某个变量重新赋值）
  • kill[S]：在表示 S 注销的数据流信息（比如在 S 中给某个变量赋新值，则其原值会被注销）

根据out[S]反推in[S]：

in[S]=(out[S]−gen[S])∪(kill[S]∪gen[S])in[S]=(out[S]-gen[S])∪(kill[S]∪gen[S]) in[S]=(out[S]−gen[S])∪(kill[S]∪gen[S])

数据流方程求解过程中的三个关键因素：

  • 当前语句产生和注销的信息取决于需要解决的具体问题，in[S]in[S]in[S] 和 out[S]out[S]out[S] 可以相互定义，正推反推均可
  • 由于数据是沿着程序的执行路径，也就是控制流路径流动，因此数据分析的结果受到程序控制结构的影响
  • 代码中出现的诸如过程调用、指针访问以及数组成员访问等操作，对定义和求解一个数据流方程都会带来不同程度的困难

到达定义分析

到达分析定义的目标：我们希望通过到达定义分析能获知，在程序的某个静态点 p，例如某条中间代码之前或者之后，某个变量可能出现的值都是在哪里被定义的

判断：对于在 p 处对该变量的引用，取得的值是否在 d 处被定义？

  • 我们需要检查：如果从定义点 d 出发，存在一条路径到达 p，并且在该路径上，不存在对该变量的其他定义语句，则认为变量的定义点 d 到达静态点 p
  • 而如果路径上有对该变量其他新的定义（其他赋值语句），则认为新定义会杀死或消除 d 处的旧定

到达定义分析

对于基本块中的某一条中间代码：

d1:u=v    op    wd1:u=v\;\;op\;\;w d1:u=vopw

其中 v 和 w 为变量 ，op 为操作符，代码对应的到达定义数据流方程为：

out[d1]=gen[d1]∪(in[d1]−kill[d1])out[d1]=gen[d1]∪(in[d1]-kill[d1]) out[d1]=gen[d1]∪(in[d1]−kill[d1])

其中：

  • gen[d1]={d1}gen[d1]=\{d1\}gen[d1]={d1} ：表明该语句产生了一个定义点（定义了 u），注意是定义点的集合，而不是定义的变量名的集合
  • kill[d1]kill[d1]kill[d1]：是程序中所有对变量 u 定义的其他定义点的集合，包括在 d1 之前和 d1 之后的所有对变量 u 定义的定义点（之所以要把 d1 之后的定义也注销，是因为程序执行可能会有跳转 / 循环等打破顺序结构的语句）
  • 对于改代码在同一基本块内紧邻的后继代码，假设其为 d2，则有 in[d2]=out[d1]in[d2]=out[d1]in[d2]=out[d1]，也就是上一条语句的输出等价于下一条语句的输入

活跃变量分析

活跃变量分析

（Live-variavle Analysis）到达定义分析是沿着流图路径的，而有的数据流分析是反方向计算的

活跃变量分析：变量 x 在某个执行点 p 是活跃的，即变量 x 的值在 p 点或沿着从 p 点出发的某条路径中会被使用，则称 x 在 p 点是活跃的

  • 通过活跃变量分析，可以了解到某个变量 x 在程序的某个点上是否活跃，或者从该点出发的某条路径上是否会被使用，如果存在被使用的可能，x 在该程序点上就是活跃的，否则就是非活跃，或者死的

活跃变量的数据流分析方程

数据流方程：

in[B]=use[B]∪(out[B]−def[B])in[B]=use[B]∪(out[B]-def[B]) in[B]=use[B]∪(out[B]−def[B])

这是个从后往前推导的过程，也就是说我们要用这个基本块的 out，去这个它的 in：

  • def[B]def[B]def[B]：在基本块 B 中，在使用前被定义的变量集合，在引用该变量前已经明确地对该变量进行了赋值
  • use[B]use[B]use[B]：在基本块 B 中，在定义前被使用的变量集合，在该变量的任何定义之前对其引用

直观来讲，我们要从 B 块给它上面的基本块传递一个信息，这个信息表示有哪些变量是在上面那些基本块中被定义，然后在 B 以及它下面的基本块被用到了，这些变量就是所谓活跃变量，因此我们需要考虑三部分组成：

  • out[B]out[B]out[B]：在 B 下面的那些基本块被用到的变量，有可能是在 B 上面的基本块中被定义过
  • def[B]def[B]def[B]：如果一个变量在 B 中被重新定义，那么这个新定义就会覆盖上面的老定义，因此上面这个变量相应就失效了，变成死变量，因此需要减去，即后面这个变量所有别引用的地方所参考的都是当前这个位置的定义
  • use[B]use[B]use[B]：在 B 这个基本块中被使用且没有在使用前被定义的变量，一定是来自 B 上面基本块中的定义

也就是说，如果在路径后方的某个基本块中，变量 x 被使用，则沿着执行路径的逆向直到 x 被定义的基本块，x 都是活跃的

通过这三部分，我们加工出来一个 in[B]in[B]in[B]，就表示 B 这个基本块上面那些基本块中的活跃变量

要想计算每个基本块的 ininin 和 outoutout 函数，我们需要先独立计算每个基本块的 useuseuse 和 defdefdef 函数，我们以下面这张图为例进行分析：

其中我们需要再次强调 def 和 use 的定义：

  • defdefdef：变量在基本块中的定义（赋值）先于任何对它们的使用
  • useuseuse：变量在基本块中的使用先于任何对它们的定义（赋值）

这里需要注意的就是形如 x=x+1x=x+1x=x+1 这样的定义点，这里显然是先引用 x 的值，再重新给 x 赋值，因此是先引用后定义，x 仅属于 useuseuse 集合，而不属于 defdefdef 集合

基本块的活跃变量数据流分析

输入：程序流图（由基本块组成），且每个基本块的 useuseuse 集合和 defdefdef 集合都已经计算完毕

输出：每个基本块入口和出口处的 ininin 和 outoutout 集合

  • 根据以下两个方程 out[B]=∪B的后继基本块Pin[P]out[B] = ∪_{B的后继基本块P}in[P]out[B]=∪B的后继基本块P​in[P] 和 in[B]=use[B]∪(out[B]−def[B])in[B] = use[B]∪(out[B]-def[B])in[B]=use[B]∪(out[B]−def[B]) 为每个基本块 B 依次计算集合 in[B]in[B]in[B] 和 out[B]out[B]out[B]，这里的依次指的是倒序的顺序，也就是从 exit 块倒序往前求，如果计算得到的某个基本块的 in[B]in[B]in[B] 和此前计算得出的该基本块的 in[B]in[B]in[B] 不同，则循环执行上面两个公式，直到所有基本块的 in[B]in[B]in[B] 集合都不再产生变化为止

有关变量冲突判断的算法

判断核心：两个变量中的一个变量在另一个变量定义（赋值）处是活跃的，则他们就是冲突的

算法1：在每一个变量的定义点计算活跃变量

算法2：计算基本块入口处的活跃变量（ininin 的集合），这些变量在该基本块中的定义点活跃，因而冲突，之后在基本块内部，进一步计算每个定义点的活跃变量（这些活跃变量选择该基本块的 ininin 集合），这样能降低计算的复杂度，因为基本块内部是线性的

循环优化

（1）循环不变式的代码外提：

循环中的不变表达式指的是不随循环控制变量的改变而改变的表达式或子表达式

（2）循环展开：

循环展开用于把构成循环体的代码（不包括控制循环的测试和转移部分）重新产生许多次，而不仅仅是一次，是一种用空间换时间的优化方式

在使用循环展开作为循环优化的时候，需要注意以下问题：

（3）归纳变量优化：

归纳变量指的是在每一次循环迭代的过程中，如果某个变量的值固定增加或减少一个常量值，就称该变量为归纳变量（induction variable），即若当前执行循环的第 j 次迭代，归纳变量的值应为 c∗j+c′c*j+c'c∗j+c′，这里 c 和 c‘ 都是常量

（4）in_line 展开：

把过程 / 函数调用改为 in_line 展开可以节省许多处理过程 / 函数调用所花费的开销，常见的比如int max(a,b){return a>b?a:b;} 可以不写成外部函数的形式，而是把函数体内容直接作为一条语句嵌入到调用位置

这样可以省去函数调用时的参数压栈，保存返回地址等指令，不过也仅限于简单的函数
