错误处理

错误处理概述

编译程序对于语法和语义正确的源程序，要能够正确编译产生等价的目标代码，而对于错误的源程序不能发现一个错误就立刻停止，而是要能检查出错误的性质和出错位置，并且使得编译能够进行下去，同时尽可能多而准确的发现和指出各种错误

错误分类

从编译角度，错误可以分成语法错误和语义错误两类：

  • 语法错误：程序结构不符合语法规则或者词法规则的错误
  • 语义错误：程序不符合我们所编写的语言的语义规则（上机实验中处理的语义错误）或超越具体计算机系统的限制（数据溢出错误，常数太大，计算结果溢出；符号表，静态存储分配数据区溢出；动态分配数据区溢出等）

错误诊察

违反语义和语法规则以及超过编译系统限制的错误，由编译程序在语法分析和语义分析过程中诊察出来（其中语义分析需要借助符号表）

下标越界，计算结果溢出，动态存储数据区溢出等错误在目标程序运行时才能检测，因此需要由目标程序诊察

因此，编译程序需要生成相应的目标程序，对运行代码进行检查并处理

错误处理

发现错误以后，在报告错误的同时，还要求对错误进行处理，以方便编译能够继续进行下去，目前有两种处理方法：

  • 错误改正：编译发现错误以后，根据文法对错误进行改正（对于简单的语法错误，如缺少数组括号，分号等可能容易改正，但是对于较为复杂的错误，如运算符号冗余，表达式缺少括号等则难以改正）
  • 错误局部化处理：当编译程序发现错误以后，尽可能把错误限制在一个局部的范围内，避免错误扩散和影响程序其他部分的分析

错误的局部化处理

一般原则：当诊断到错误以后，就暂停对后面符号的分析，跳过错误所在的语法成分（一旦跳过就认为该语法成分是正确的），然后继续向下分析

  • 词法分析：发现不合法字符，显示错误，并且跳过该标识符 / 单词，继续向下分析
  • 语法语义分析：跳过所在的语法成分（短语或语句），一般是跳到语句的右界符（一般就认为是一条语句的结尾 \n 处），然后从新语句继续向下分析递归下降分析法

如果我们定义右界符为 \n，可能会导致跳过的符号过多，会漏判一些错误信息，因此我们需要提高错误局部化程度，我们维护两个集合：

  • S1 集合：合法后继符号集合（某语法成分的后继符号）
  • S2 集合：停止符号（跳读必须停止的符号集合）

当发现错误的时候，我们重复跳过字符，直到读取到 S1 或 S2 集合中的符号，就停止跳读（相当于两个集合取并集）

目标程序运行时错误的检测和处理

在编译时，我们需要生成检测这类错误（下标变量下标值越界，计算结果溢出，动态存储分配数据区溢出）的代码，而对于这类错误，要正确报告出错误位置很困难，因为目标程序与源程序之间很难建立位置上的对应关系

一般处理方法：当目标程序运行检测到这类错误时，就调用异常处理，打印错误信息和运行现场（寄存器和寄存器中的值），然后直接停止程序运行即可

遏制重复的错误信息

语义错误主要来源于标识符未声明和类型不一致等错误，这种错误往往会多次出现（比如有一个未声明的变量 x，每次使用都会触发变量未声明错误）

一般处理办法：建立一张错误名字表，每次发现出错名字后，先查表，若先前已经报过这个错误，则不再打印错误信息，否则登记到表中，并且打印错误信息

溢出错误：即编译程序所需的工作区或源程序所需的数据区域溢出，为了使得编译程序正常运行，同样需要遏制重复的错误信息

一般处理方法：对各数据区分别设置一个标志单元，初始为 0，当发生溢出时，先查看单元内容，若为 0，则表示刚发生溢出，打印错误信息，把标志设为 1；若为 1，则不再重复打印

用来实现上述逻辑的代码需要在目标代码运行时发挥作用，它同样是由编译程序所生成的（编译程序在生成目标码的同时，也会生成和目标码运行时错误处理有关的代码）

语法制导翻译技术

语义分析的过程和语法成分密切相关，我们每识别到一个完整的语法成分，都需要调用符号表，进行查表 / 写入操作，这意味着我们的语义分析需要在语法成分识别的框架下完成，因此语义分析又称为语法制导的翻译技术

严谨点来讲全称应该是：语法制导下的语义分析与翻译技术

编译中我们所关心的程序语义是操作语义，也就是和程序怎么执行 / 运行之间的关系

输入文法和翻译文法

输入文法：未插入动作符号的文法，由输入文法可以通过推导产生输入序列

翻译文法：插入动作符号的文法，由翻译文法可以通过推导产生活动序列，而活动序列由输入序列和动作序列两部分所组成

我们分别用原文法和加入动作符号后的文法做一遍推导，把 @ 当成终结符

对于活动序列，我们可以分别得到输入序列和动作序列两部分，其中动作序列就是我们需要的后缀表达式，我们依次执行动作序列中的动作，打印出来的就是后缀表达式

活动序列 = 输入序列 + 动作序列

翻译文法是上下文无关文法，终结符号集合由输入符号和动作符号组成，由翻译文法所产生的终结符号串称为活动序列，翻译文法和原文法相比终结符集合发生了变化，加入了（动作符号+字符串）组成的若干新终结符

这种带有动作符号的翻译文法也称为符号串翻译文法，输入文法中的动作符号对应的语义子程序是输出动作符号标记 @ 后的字符串的文法（在输入文法中加入一些新文法，这些文法由动作符号标识，作用为输出 @ 后的字符串）

语法制导翻译：按照翻译文法所进行的翻译，给定一个输入符号串，根据翻译文法获得翻译该符号串的动作序列，并且执行该序列所规定的动作过程

语法制导翻译的实现方法

在文法的适当位置插入语义动作符号，当按照文法分析到动作符号时，就调用相应的语义子程序，完成翻译任务

翻译文法所定义的翻译是由输入序列和动作序列所组成的对偶集，即给定了一个翻译文法，就相当于给定了一个对偶集

补充：

翻译文法就是在上下文无关文法的产生式中插入动作（如打印、生成代码、压栈、输出符号等）的文法
当我们对某个输入串进行一次合法的推导 / 分析时，就会得到一条动作序列（把所有插入在产生式右部的动作按实际触发顺序依次串起来）
因而，“由翻译文法定义的翻译” 可以看成是一组二元组 (输入序列, 动作序列) 的集合：每个合法输入串，对应着按该文法推导时触发的那条唯一的动作序列

对偶集：输入序列和动作序列一一对应，因为一但文法和动作都固定下来，对于每个能被该文法接受的输入序列，分析过程会唯一确定动作触发的位置和次序，也就唯一生成了与之配对的动作序列

属性翻译文法

属性翻译文法

翻译文法 -> 属性文法

在翻译文法的基础上，我们进一步定义属性文法，翻译文法中的符号，包括终结符，非终结符和动作符号均可带有属性，这样能够更好地描述和实现编译过程

直观来讲：属性就是贴在语法树每个结点（符号）上的**“小纸条 / 字段”**，用来携带 / 传递语义信息或中间结果，动作符号，终结符，非终结符都可以拥有这种用于传递信息的字段

字段里的内容可以是任意你在翻译 / 编译时需要的内容：类型、值、地址、生成的目标代码片段、回填链表、行号、作用域信息、是否为左值、常量性、数组维度……甚至是 “函数指针 / 过程” 本身

属性可以分成两种：

  • 综合属性：语义函数的返回值（语法树上从子节点往父节点汇总的信息）
  • 继承属性：语义函数的参数（语法树上从父节点或左兄弟传下去的信息）

综合属性

其中对于终结符的综合属性是我们直接赋予（可能是外部输入）的（比如 i + i 这个表达式，我们直接令 i 分别等于 3 和 5，相当于人为赋值），而对于非终结符的综合属性，则可以通过画出语法树加自底向上推导的形式，确定每个非终结符的综合属性

每个非终结符的综合属性来自它子的节点的汇总

也可以写成一条一条推导式的形式，然后通过引入属性变量名，借助求值规则来实现综合属性的推导

总的来讲，综合属性的推导，如果是使用语法树进行推导，则为自底向上；如果是使用推导式进行推导，则为自右向左

继承属性

继承属性和综合属性恰好相反，继承属性的推导是自左向右，自顶向下，即从产生式的左侧获取信息，传递给右侧的产生式

假设我们有语句int A, B;，我们在填表的时候，先填入变量类型为 int，再填入变量名 A，当我们要登记变量名 B 的时候，其中标识符 B 是综合而来的（键盘输入），而 B 的变量类型则是继承而来的，由先前定义的 A 的类型所决定（这是由逗号分隔定义的变量类型相同而决定的）

  • 当我们定义形如 int A, B, C 这样的多个变量时，当处理到后边的变量 B C 时，我们需要知道变量的类型

首先对于终结符 Type 和 id，它们的有关信息可以通过词法分析得到（即相当于从外界直接获取输入），我们为其设置两个综合属性：

  • $Type_{\uparrow t}$：t 表示变量类型
  • $id_{\uparrow n}$：n 表示变量名

因此填表的动作符号 $@set_table$ 和变量表也需要得到这些属性信息，此时它们必须从 Type 或 id 处拿到这些属性，对于这种从前面符号处得到的属性，就称为继承属性，继承前面符号的值：

  • $@set_{table\downarrow t_1,n_1}$：从 Type 和 id 处继承得到
  • $<var;; table>_{\downarrow t_2}$：从 Type 处继承得到

  • 输入文法
  • （字符串）翻译文法：加入动作符号
  • 属性翻译文法：加入综合属性和继承属性，借助求值规则实现

属性翻译文法的自顶向下翻译

（1）L-属性翻译文法（L-ATG）

这是属性翻译文法中比较简单的一种，其输入文法要求是 LL(1) 文法，可以用自顶向下分析方法构造分析器，在分析过程中可以进行属性求值

特征：每个符号的继承属性只依赖于该符号左边的信息

L-属性翻译文法是带有下列说明的翻译文法：

  • 文法中的终结符，非终结符以及动作符号都带有属性，且每个属性都有一个值域
  • 非终结符号以及动作符号的属性可以分为继承属性和综合属性（同时有入参和返回值）
  • 开始符号的继承属性具有指定的初始值
  • 输入符号（终结符号）的每个综合属性具有指定的初始值
  • 属性值的求值规则如下所示

继承属性求值规则

继承属性——体现自顶向下，自左向右的求值特性：

  • 产生式左部的非终结符号的继承属性值，取前面产生式右部同名符号已有的继承属性值
  • 产生式右部符号（非终结符，动作符号）的继承属性值，用该产生式左部符号的继承属性或出现在该符号左部的符号的所有属性值进行计算
  • 对于第二条规则，产生式左部（->符号左侧）和该符号左部是两个不同的概念，前者只能用其继承属性，后者可以同时用其继承和综合属性

综合属性求值规则

综合属性——体现自底向上，自右向左的求值特性：

  • 产生式右部非终结符号的综合属性值，取其下面 / 后面产生式左部同名非终结符号的综合属性值
  • 产生式左部非终结符号的综合属性值，用该产生式左部符号的继承属性或该产生式某些右部符号的任意属性进行计算
  • 动作符号的综合属性用该符号的继承属性或该符号右部某些符号的任意属性进行计算
  • 注意区分，产生式左部非终结符对应的是产生式左部符号（包括该符号本身，只要是在产生式左部的符号即可）的继承属性；而动作符号对应的是该符号本身的继承属性

求值顺序规则

属性值的求值顺序：$A\rightarrow B C$

求值顺序为：

  • A继承：来自该式前面表达式右部符号 A 的继承属性
  • B继承：来自 A 的继承属性
  • B综合：来自该式后面表达式左部符号 B 的综合属性
  • C继承：来自 A（表达式左部）的继承属性和 B（符号 C 左部）的全部属性
  • C综合：来自该式后面表达式左部符号 C 的综合属性
  • A综合：来自 A 本身的继承属性和 B C 中某些符号的综合属性

注意：

  • 终结符只有综合属性（由词法分析器提供）
  • 非终结符和动作符号既可以有综合属性也可以有继承属性
  • 文法开始符号的所有继承属性作为属性计算前的初始值

一般来说，对于出现在产生式右边的继承属性和出现在产生式左边的综合属性都必须提供一个求值规则，属性求值规则只能使用相应产生式中的文法符号和属性（这有助于在产生式范围内封装属性的依赖性），但出现在产生式左边的继承属性和产生式右边的综合属性不由所给的产生式的属性求值规则进行计算，它们由其他产生式的属性规则计算

求值顺序：$A \rightarrow BC$

  • A 的继承属性：若 A 为开始符号，则有指定值；否则由上面产生式右部符号 A 的继承属性求得
  • A 的综合属性：由 A 的继承属性或产生式某些右部符号的所有属性求得
  • B 的继承属性：由 A 的继承属性求得
  • B 的综合属性：由下面产生式中左部符号 B 的综合属性求得
  • C 的继承属性：由 A 和 B 的继承属性求得
  • C 的综合属性：由下面产生式中左部符号 C 的综合属性求得

注意：总结符号只能有综合属性（由词法分析器提供），而非终结符和动作符号可以同时有继承属性和综合属性

这里倒不是说综合符号就不能有继承属性，只是没有必要，因为综合符号在翻译过程中就相当于叶子节点，其信息由词法分析阶段决定，词法分析一旦把输入切成 token，像 ID.lexeme、NUM.value、tok.line 等信息就固定了，这些就是终结符的综合属性，继承属性的作用是“把父节点 / 左兄弟的上下文传给子结点以指导展开 / 翻译，而终结符不再展开，没有下游要受它影响，因此没必要传继承属性下来

简单赋值形式的L-属性翻译文法

属性翻译文法是 L-ATG，简单赋值形式的属性翻译文法是 SL-ATG

SL-ATG

一般情况（普通的 L-ATG）： x := f(y, z) 我们称 x 的属性值是 y 和 z 的属性值的函数

而在 SL-ATG 中： x := 某符号的属性值或常量，比如我们有x:=y或者x,y,z:=17，这称为复写规则

为了实现上的方便，我们希望文法符号的属性求值规则为上述简单形式（也就是 SL-ATG 形式），为此，我们对现有的 L-ATG 的定义做一点改变，从而形成一个称为简单赋值形式的 SL-ATG

（定义）一个 L-ATG 被定义为简单赋值形式的 SL-ATG，当且仅当满足以下条件：

先统一记号：一条产生式写成 $A \rightarrow X_1,X_2,\cdots,X_n$

  • 左部符号＝A（产生式箭头左边的那个非终结符）
  • 右部符号＝右边序列里的每一个 $X_i$（它们可以是非终结符或终结符）

在这个前提下，定义需要满足的两个条件（SL-ATG：简单赋值形式的 L-属性翻译文法）可精确表述为：

条件 1（右部符号的继承属性怎么赋值）

对右部中每个符号 $X_i$ 的继承属性 $X_i.a$，其赋值只能是简单赋值，且取值来源仅限于三类：

  1. 一个常量：Xi.a := c
  2. 左部符号 A 的某个继承属性：Xi.a := A.b
  3. 右部中位于 $X_i$ 之前的某个符号 $X_j$（j<i）的综合属性：Xi.a := Xj.c

也就是说，$X_i$ 的继承属性不能依赖自己或右边符号的属性，只能看A和左侧兄弟，这是 L-属性的关键约束，保证可自左向右一次求值、无环

条件 2（左部符号的综合属性怎么赋值）

对左部非终结符 A 的综合属性 $A.a$，其赋值也必须是简单赋值，且取值来源仅限于：

  1. 一个常量：A.a := c
  2. A 自己的某个继承属性：A.a := A.b
  3. 右部中某个符号 $X_k$（k 可为 1..n）的综合属性：A.a := Xk.c

因为在规约 / 返回父结点时，右部各子结点的综合属性都已经就绪，所以 A 的综合属性可以取自任何一个子结点的综合属性，或直接沿用 A 的继承信息，或设为常量

目的：一个简单赋值定时的 L-ATG 除动作符号外，其余符号的属性求值规则（就是形如 x:=y 这样的赋值表达式）右部是属性或者常量（简单化）

也就是说在简单赋值形式的 L-属性翻译文法（SL-ATG）里，除了 “动作符号” 可以做任意过程外，其他一切属性求值都必须是简单赋值，也就是赋值表达式的右边只能是：

  • 某个属性：X.a := Y.b（把别处已有的属性直接拷贝过来）
  • 一个常量：X.a := c

不允许写成带计算 / 组合/函数调用的复杂式子，例如：

  • ⛔ X.a := Y.b + Z.c
  • ⛔ X.a := f(Y.b)
  • ⛔ X.a := Y.b || Z.c || "*"（字符串拼接）

如果需要做 “复杂工作”（打印、拼接、生成代码片段、回填等），应放到动作符号里（如 @emit("+")、@genCode(...)），动作符号可以执行过程，但属性间的赋值必须保持 “直接拷贝已有属性或赋值常量”

L-ATG 到 SL-ATG 的转化

给定一个 L-ATG，考虑如何将其转化为等价的赋值形式的 SL-ATG，我们以 $\rightarrow a_{\uparrow R}{\uparrow S}{\downarrow I}$ 为例，这里 $I:=f(R,S)$，此时显然继承属性的求值规则不是简单赋值形式的，因为它需要对函数 $f$ 进行运算

我们考虑通过以下步骤的变形，变为简单赋值形式的 L-ATG：

  • 设动作符号 $@f$ 表示函数 f 求值，该动作符号对应两个继承属性 I1 I2 和一个综合属性 S1，即 $@f_{\downarrow I_1,I_2 \uparrow S1},S1:=f(I_1,I_2)$，我们可以认为 I1，I2 是入参，S1 是返回值
  • 修改产生式，插入 $@f$ 到产生式右部的适当位置，引入新的复写规则（把 R, S 赋值给 I1, I2，同时 f 的运算结果赋值给 S1），最后删去原有的包含 f 的规则 S1:=f(I1,I2)
  • 最后得到的简单赋值形式的 L-ATG 如下：

$$
\rightarrow a_{\uparrow R}{\uparrow S}@f{\downarrow I_1,I_2 \uparrow S_1}_{\downarrow I}\
I_1:=S,I2:=R,S1:=f(I1,I2),I:=S1
$$

相当于就是我们不想要这种运算函数 f，但是从 R 和 S 拿到 I 又必须经过这个函数计算，前面我们又提到过动作符号里可以放任意复杂的运算函数，因此我们就在 R S 和 S1 之间插入一个代表这种运算的动作符号，实现一个承接 / 桥梁的功能
