分块

分块原理

假设数据规模为 n，我们尽可能使得划分出来块的数量和每一个块的大小尽可能相同，比如都是 √n 的规模

使用分块结构维护信息时，散块的块内信息进行暴力维护，整块的块内信息直接修改标签即可，因此每次处理区间修改或区间查询时，最多处理左侧散块+右侧散块+中间若干整块即可

块内遍历的复杂度为 O(√n)，块间遍历的复杂度也为 O(√n)

在分块的模版中，我们通常需要维护以下几个量：

  1. bi [ i ]：原数组 arr 中第 i 个下标所属的分块的编号
  2. bl [ i ]：第 i 个块的左边界
  3. br [ i ]：第 i 个块的右边界（注意 1 和 2,3 中 i 的含义不同）
  4. blen：分块的长度
  5. bnum：分块的数量

单点修改 & 区间查询

SP18185

【区间内不小于某个值的数字个数的查询】

思路

通过本例，我们介绍使用分块维护区间内不小于某个值的数字个数时，通常使用的维护思路：我们先把原数组 arr 拷贝一份得到 sorted，并且对 sorted 数组以块为单位进行排序，也就是在每个块内，保证 sorted 数组是有序的

单点修改：我们只需要修改对应 arr 数组位置的值，然后把这个位置所属的块整体重新刷新到 sorted 数组中，并且对 sorted 数组重新排序即可，整体复杂度为 O(√n + √nlog√n)

区间查询：我们可以把查询分成三部分，左边不完整块中的若干散点，中间若干个整块，右边不完整块中的若干散点，其中左右的散点可以直接通过遍历 arr 本身暴力得到，而中间的每个块中，sorted 数组已经有序，我们查询其中不小于 key 值的数字的个数，可以用二分查找进行加速，整体复杂度为 O(√n + √nlog√n)

因此整体修改+查询的复杂度即为 O( m×(√n + √nlog√n) )，而分块的常数非常小，因此可以极限卡过本题

P2801

【区间内不小于某个值的数字个数的查询】

思路

本例维护的总体思路和上例一样，对于散块进行暴力查询，对于整块维护有序的 sorted 数组进行二分查找，现在我们考虑对区间修改的处理

区间修改：可以考虑线段树的处理思路，维护一个 lazy 数组，用来表示对每个整块所做的累计操作，lazy [ i ] = j 表示第 i 个块在先前的区间修改中，总共加上了 j 的偏移量

那么我们的区间修改处理也可以仿照区间查询的思路，分成左右的散块和中间的整块分别进行处理，对于左右散块的修改，我们直接暴力遍历 arr 数组对应的位置直接进行修改，然后把修改后的左右的两个块完整地刷回到 sorted 数组中，并且进行重新排序，而对于中间整块的修改，我们直接把修改记录在 lazy 数组中即可，这样我们整体修改的复杂度也是 O(√n + √nlog√n) 的

区间查询：和上例完全相同，只是要考虑 lzay 数组的影响，每次查找某个整块或散块中不小于 v 的数字时，需要先找到其块号 i，然后另 v - lazy [ i ] 先消除区间修改的影响，再进行查询（其实可以类比成懒标记的下发）

P5356

【分块暴力查询区间第 k 小】

思路

我们已经实现了对区间中，不小于 v 的数字的个数的查询，接下来我们考虑再此基础上实现本例的要求

本例要求我们查询区间第 k 小值，我们不妨假设我们找到了一个值，然后我们检验这个值是不是第 k 小，也就是检验是否有恰好 k 个数不超过它，也就是如果我们有了一个 v，我们查询不大于 v 的数字的个数，那么本例就转化成上例已经实现过的查询了

至于我们假设找到的那个值是什么，我们显然可以通过二分答案的方式求出，而我们上面的这段分析则恰好是 check 函数所要实现的功能，所以对于本例的查询，我们首先需要知道我们二分答案的范围，也就是给定 l 和 r 我们要查出区间上的最大值和最小值，这两个查询直接仿照上例即可实现，然后我们在 maxv 和 minv 之间二分答案，对于每次获得的一个 v 值，我们都检查在 l 到 r 区间上是不是至少有 k 个数字比他小

具体检查的过程，就又变成了区间上不超过 v 的数字个数的查询，使用上例的思路即可，总体上来看本例代码量还是挺大的，尤其是 bi bl br 随便哪个下标写错会很难发现，我在实现的时候也是调了很久才通过（

P3870

【分块维护区间翻转】

思路

本例是难题也是好题，通过本例可以强化分块中一个很重要的性质：我们对散块的修改直接作用在原本的 a 数组上，对整块的修改作用在我们额外对每个块维护的 tag 数组上，这两部分修改是互相独立的，我们只有在最后统计答案的时候，才会对这两部分影响进行叠加

对于本例，我们需要额外维护这样几个信息：

  1. sum [ i ]：第 i 个块上，不考虑 tag 的标记，仍然开着的灯的数量
  2. tag [ i ]：第 i 个块上的灯，整体开关情况经过翻转的次数

翻转操作：我们还是把区间拆成左右散块和中间整块来处理，对于中间整块没什么好说的，对每个块的 tag 01 翻转即可，因为翻转两次相当于没反转，我们甚至开一个 bool 类型数组都可以表示，而对于两侧的散块而言，我们只需要直接修改每个位置 a 的值，让每个单点进行 01 翻转即可，这里需要注意，翻转完成以后，我们需要把发生翻转块的 sum 值重新记录，但是只记录 a 值即可，不需要考虑 tag，比如 tag == 1 时，但某个块的状态为 100111，我们还是记录这个块的 sum 值为 3 而不是 2

查询操作：查询操作也拆成左右散块和中间整块来处理，对于左右散块，我们此时需要结合它自己的 a 值和这个点所在整块的 tag 值，而这个共同作用贡献每一个单点的值；而对于中间的整块，我们需要根据 tag 值来确定开着的灯的个数，如果 tag 值为 1，说明整体做过翻转，我们需要统计 blen - sum 的值，如果 tag 值为 0，说明整体没做过翻转，我们统计的 sum 值就是实际开着的灯的数量

这里比较关键的就是用 a 值更新 sum 值的时候，是不需要考虑 tag 的情况的，也就是说我们允许 sum 维护的不是最终答案，只要它能结合 tag 和 blen 推出最终的正确答案即可，换言之，对于分块而言，tag 标记是不需要随时下发的，这也是分块和线段树相比，好写的地方

AcWing - 2166

【分块维护跳跃问题】

思路

本例的模型是序列上的每个点都有一个向后跳跃的步数 a [ i ] 表示从 i 位置可以跳到 i + a [ i ] 位置，问给定一个起点，从该起点一直向后跳，最少跳多少次能够跳出 n 的范围

如果我们暴力求解，最坏的情况就是 a 数组全是 1，那么跳跃的复杂度几乎是 O(n) 级别，因此结合分块我们很容易想到，我们没必要一个点一个点的跳，可以一个块一个块的跳，也就是维护对于每个位置 x，它跳出自己所在的那个块，所用的最少步数

这里我们维护的信息并不是以块为整体维护的，因为对于块内的不同点而言，其跳出块的步数和最终落点势必不同，这里我们只是借助分块的思想，提前预处理出来每个位置跳出自己所在的那个块的最终落点和最小步数，因此我们需要维护两个数组：

  1. step [ i ] : i 位置跳出它所在的块所需的最小步数
  2. jump [ i ] : i 位置跳出它所在的块的最终落点坐标

这两个数组可以在 O(n) 的复杂度内构造，因为我们可以倒序遍历，j 位置的信息可以通过 j + a [ i ] 的信息得以推导，具体来讲，如果 j + a [ i ] 能够跳出当前块，就直接更新 step [ j ] = 1，反之就更新 step [ j ] = step [ j + a[j] ]，依次类推

那么对于单点修改，我们只需要改完单点的 a 值以后，再更新一遍这个块内所有的 jump 和 step 值即可，更新方式和当前块的初始化方式相同；而对于单点查询，我们只需要借助 jump 数组不断向后跳，直到其跳出 n 的范围即可

最后，本例的下标都是从 0 开始的，而我们分块一般下标是从 1 开始，整体做一次输入平移即可

前缀信息维护

在需要维护前缀信息的分块类问题中，我们一般会额外定义下面三个数组，辅助我们对整体块信息和散块信息的统计：

  1. freq [ i ] [ j ]：前 i 个整块中，数字 j 出现的次数
  2. key [ i ] [ j ]：从第 i 个块到第 j 个块之间，序列满足的某种性质（一般和要维护的性质有关）
  3. numCnt [ i ]：在遍历散块的时候，数字 i 出现的次数

下面的题目中，我们会结合这些信息进行统计

P4168

【区间众数的查询】

思路

本例只有查询，没有修改，题意经过转化后，就是查询区间 l 到 r 范围内的众数，并且强制在线查询（如果是离线查询，可以用莫队解），这里如果强制在线的话，我们考虑用以上信息辅助分块求解：

  1. freq [ i ] [ j ]：前 i 个整块中，数字 j 出现的次数
  2. mode [ i ] [ j ]：从第 i 个块到第 j 个块之间的众数
  3. numCnt [ i ]：在遍历某个散块的时候，数字 i 出现的次数（每次遍历某个散块之前，都要清空对应的 numCnt 数组，该数组是支持复用的）

假设我们已经有了 freq 和 mode 数组的信息，那么对于一段 l 到 r 范围内众数的查询，我们可以先找到中间的那些整块的范围，假设是第 ll 到第 rr 块，那么我们可以通过查询 mode 中的信息，求出这些整块中的整体的众数，接下来我们再分别遍历两次左右两个散块中的数字，第一次把他们额外的出现次数计入 numCnt 数组中，第二次我们再遍历一遍两个散块中的数字，考虑他们的出现次数是否能超过我们在整块中找到的众数，也就是某个散块中的数字 x 的 numCnt 加上它在整块中的出现次数（可以用 freq 前缀和作差得到）是否能超过当前众数的 numCnt 加上它在整块中的出现次数（同理用 freq 求解），这样我们就可以在 O(√n) 的复杂度内完成一次查询

接下来只需要考虑 freq 数组和 mode 数组的求解：

  1. freq 数组：考虑前缀思想，对于第 i 个块，我们先遍历第 i 个块中的每个数字，统计他们出现的次数，接着我们再遍历所有可能出现的数字，让它对应第 i 个块中的出现次数加上前 i-1 个块中的出现次数
  2. mode 数组：同样考虑前缀思想，对于 i 到 j 块内的众数，我们先假定它和 i 到 j-1 块内的众数相同，接着我们还是额外统计第 i 个块中每个数字出现的次数 numCnt，然后把这个 numCnt 和前 i 到 j-1 个块中这个数字出现的次数相加，和当前我们假定的众数的总出现次数向比较，考虑能否更新新的众数

最后还有一个问题，每个位置的数字 a 的范围到 10^9^，下标数组肯定开不下，但是最多只有 10^4^ 个数，所以我们考虑先进行离散化以后即可开下

P4135

【区间内出现偶数次的数字个数的查询】

思路

本例和上例十分类似，都是求解区间中出现次数满足某种性质的数（众数是出现次数最多，本例是出现次数为偶数），他们有一个共同的性质，都是可以由子状态线性推导得出的，无非是众数的出现次数只会单调不减，但出现次数是偶数的性质会 01 摆动，但是他们都可以由当前块扩出一个新块，在扩展的过程中完成更新，因此我们同样可以考虑用对以块为单位的前缀思想进行解决

我们还是维护以下三个数组：

  1. freq [ i ] [ j ]：前 i 个整块中，数字 j 出现的次数
  2. even [ i ] [ j ]：从第 i 个块到第 j 个块之间出现次数为偶数的数字的个数
  3. numCnt [ i ]：在遍历某个散块的时候，数字 i 出现的次数（每次遍历某个散块之前，都要清空对应的 numCnt 数组，该数组是支持复用的）

我们先考虑 freq 和 even 数组的更新，其中 freq 的更新思路完全不变，而 even 数组的更新也更加简单，对于从 i 到 j 范围内的块对应的 even 数组，它可以由 i 到 j-1 范围内的块对应的 even 数组结合第 j 个块中的数字出现情况推导得出，我们依次遍历第 j 个块内的数字 x，统计当前它出现的次数（这里的当前指的是不算 x 这一次出现，之前 x 总共出现的次数），也就是 numCnt 中的值和 freq 使用前缀加工出的整块中出现的次数，根据其出现次数不同，我们可以分三种情况讨论：

  1. x 之前出现了 0 次，再获得一个 x，对出现次数为偶数的数字个数无影响
  2. x 之前出现了奇数次，再获得一个 x，出现次数为偶数的数字个数+1（当前多了 x 这个数）
  3. x 之前出现了偶数次，再获得一个 x，出现次数为偶数的数字个数-1

我们对于第 j 个块中遍历到的每个数字，都微调 even 数组的值，即便某个 x 在第 j 块中出现了多次，之前出现带来的影响也会被加减抵消，我们即可更新出 even 数组的值

最后有了这两个数组，我们即可把 l 到 r 范围内的查询，处理成两个散块和中间若干整块两部分，首先我们先查询中间整块范围对应的 even 值，接着遍历两个散块，对于散块中的每个数字，我们都按照 even 数组求解的方法，对当前获得的整块中的答案进行微调，最后即可获得整个范围内的答案
