计网知识点总结

Destiny2026-04-212026-05-21

计算机网络知识点

协议与报文汇总

TCP 协议 ※

TCP 各字段详解

  • 源端口 / 目的端口（各 16 bit）：标识发送方和接收方的进程，TCP 通过IP地址+端口唯一确定一条连接（四元组）

  • 序号 Sequence Number（32 bit）：作为发送方时，本条报文段数据部分第一个字节的编号，TCP 将数据流视为字节流

  • 确认号 Acknowledgment Number（32 bit）：作为接收方时，期望收到对方下一个字节的编号（发送确认号时，要求已成功收到该编号之前的所有数据），仅当 ACK 标志位为 1 时有效。

  • 数据偏移（4 bit）：指出首部长度，单位是 4 字节（32 bit），最小值 5（20字节），最大值 15（60字节），即 TCP 数据报的首部长度范围即 20B 到 60B 之间

  • 保留字段（6 bit）：保留未用，置 0 即可，注意保留字段是 6 bit，图上标错了

  • 6 个控制标志位（各 1 bit）：

    • URG：紧急指针有效，配合紧急指针字段使用，要求某段数据立即组成报文发送，而不是在发送方缓冲区暂存
    • ACK：确认号字段有效（几乎所有报文都置 1），配合确认号字段使用
    • PSH：提示接收方尽快将数据交给应用层，而不是在接收方缓冲区暂存
    • RST：重置连接，异常终止发生时，需要先断开连接，再重新建立连接
    • SYN：请求建立连接（三次握手中用到）
    • FIN：数据发送完毕，请求释放连接

  • 接收窗口大小 Window Size（16 bit）：告知对方自己的接收缓冲区还能接收多少字节（己方接收窗口大小），实现流量控制

  • 校验和 Checksum（16 bit）：覆盖整条数据报首部 + 完整数据，还需加入 12 字节伪首部（含源 / 目的 IP 地址）一并计算，提供端到端差错检测

  • 紧急指针 Urgent Pointer（16 bit）：仅 URG=1 时有效，指出紧急数据的末尾位置

UDP 协议

UDP 各字段详解

  • 源端口（16 bit）：可选字段，若不需要对方回复可全置 0（如 DNS 客户端通常会填）

  • 目的端口（16 bit）：必填，标识接收进程

  • 长度（16 bit）：整个 UDP 数据报的字节数（首部 8 字节 + 数据），最小值为 8（仅首部，无数据）

  • 校验和（16 bit）：覆盖报文首部 + 完整数据 + 伪首部（UDP 协议的伪首部和 TCP 报文完全相同），在 IPv4 中是可选的（置全 0 表示不使用），在 IPv6 中是强制的

IP 协议 ※

IPv4 各字段详解

  • 版本（4 bit）：IPv4 固定为 4，IPv6 为 6（4 和 6 即 IP 协议版本号，无其他特殊含义）

  • 首部长度 IHL（4 bit）：同 TCP 的数据偏移，单位是 4 字节，最小值 5（20字节），最大值是 15（60字节），即 IP 报文段的首部长度范围也是 20B 到 60B

  • 服务类型 ToS（8 bit）：标识这条 IP 报文的紧急程度，如果优先级高可以优先发送，现在已不常用

  • 报文总长度（16 bit）：整个数据报（首部 + 数据）的字节数，最大 65535 字节，实际上基本达不到这么大（数据链路层要求报文数据段长度不超过 1500B）

  • 标识 Identification（16 bit）：发送方分配，同一个大数据报分成多个片时，各片共用同一标识，接收方据此重组

  • 标志 Flags（3 bit）：bit1 是 DF（Don’t Fragment，设置为 1 禁止分片）；bit2 是 MF（More Fragments，设置为 1 表示后面还有更多的分片）；bit0 保留不用

  • 片偏移（13 bit）：本分片数据在原数据报中的起始位置，单位是 8 字节

  • TTL（8 bit）：生存时间，每经过一个路由器减 1，减到 0 时丢弃，防止数据报无限转发，还可以用来统计一条数据传输链路上每一条经过的路由器信息

  • 协议（8 bit）：标识上层协议：1=ICMP、2=IGMP、6=TCP、17=UDP、89=OSPF

  • 首部校验和（16 bit）：仅覆盖 IP 首部（不含数据），因为 TTL 每跳会变，所以路由器每跳都要重新计算首部校验和

  • 源/目的 IP 地址（各 32 bit）：网络层寻址的核心字段

ICMP 协议

ICMP 各字段详解

  • 类型 Type（8 bit）：决定报文的大类，0=Echo Reply、8=Echo Request（即 ping）、3=目的不可达、11=TTL 超时等

  • 代码 Code（8 bit）：对 Type 进一步细分，例如 Type=3 时，Code=0 表示网络不可达，Code=1 表示主机不可达，Code=3 表示端口不可达

  • 校验和（16 bit）：覆盖整个 ICMP 报文（含数据部分），与 IP 首部校验和相互独立

  • 其余首部（4 字节）：随报文类型不同含义不同，Echo 报文中是 Identifier（标识 ping 会话）+ Sequence Number（序号，用于统计丢包）；差错报文中通常全置 0

  • 数据：Echo 报文携带时间戳等；差错报文中会附带触发差错的原始 IP 首部以及原数据前 8 字节（方便上层定位出错的连接）

ICMP 的差错报文

  • 目的不可达 Destination Unreachable
  • 源点抑制 Source Quench
  • 参数错误 Parameter Problem
  • 超时 Time Exceeded
  • 路由改变（重定向）Redirect

ICMP 的查询报文

  • Echo Reauest
  • Echo Reply
  • TimeStamp Request
  • TimeStamp Reply

IGMP 协议

IGMP 各字段详解

  • 类型 Type（8 bit）：区分查询 / 报告 / 离开三种报文

  • 最大响应时间（8 bit）：仅在 Query 报文中有意义，单位是 0.1 秒，告知主机在多长时间内必须响应查询（防止同时爆发大量 Report）

  • 校验和（16 bit）：覆盖整个 IGMP 8 字节报文

  • 组地址（32 bit）：D 类多播地址（224.0.0.0 ~ 239.255.255.255），Query 报文中可为全 0（表示通用查询）或具体组地址（特定组查询）；Report/Leave 中填目标多播组地址

以太网 MAC 帧 ※

以太网 MAC 帧各字段详解

  • 前导码 Preamble（7 字节）：固定为 10101010 重复 7 次，让接收方的时钟电路能跟上发送方频率（位同步），不属于帧内容，由物理层处理

  • 帧开始定界符 SFD（1 字节）：固定为 10101011，最后两个 1 告诉接收方：下一字节就是帧的真正开始

  • 目的/源 MAC 地址（各 6 字节 = 48 bit）：链路层寻址字段，目的地址的最高位为 0 表示单播，为 1 表示多播（广播地址 FF:FF:FF:FF:FF:FF 全为 1），前 3 字节是 OUI（组织唯一标识符，IEEE 分配给厂商），后 3 字节是厂商自分配的设备编号

    • 注意大部分报文都是源地址在前，目的地址灾后，以太网 MAC 帧是目的地址在前，源地址在后

  • 类型 EtherType（2 字节）：标识上层协议：0x0800 = IPv4，0x0806 = ARP，0x86DD = IPv6，0x8100 = 802.1Q VLAN 标记（≥1536 时为 EtherType；≤1500 时为 IEEE 802.3 格式中的长度字段）

  • 数据 + 填充（46 ~ 1500 字节）：最小 46 字节是为了保证帧总长度不低于 64 字节，这是 CSMA/CD 碰撞检测的要求，帧太短的话，发送方在还未完成发送时就已收到碰撞信号，无法检测到，最大 1500 字节即以太网 MTU

  • 帧校验序列 FCS（4 字节）：使用 CRC-32 算法，覆盖目的 MAC 到数据字段，由接收方重新计算并与 FCS 比对，不一致则丢弃帧

ARP 协议

ARP 各字段详解

  • 硬件类型（16 bit）：指定链路层协议，以太网固定为 1。

  • 协议类型（16 bit）：指定网络层协议，IPv4 固定为 0x0800，与以太网帧的 EtherType 值一致。

  • 硬件地址长度（8 bit）：MAC 地址字节数，以太网为 6。

  • 协议地址长度（8 bit）： IP 地址字节数，IPv4 为 4。

  • 操作码（16 bit）：1 = ARP Request（请求），2 = ARP Reply（应答）。还有 3/4 用于 RARP（反向 ARP，已废弃）。

  • 发送方 MAC / IP（各 6 / 4 字节）：发送这个 ARP 报文的主机自身的地址，请求和应答中都填写真实值。

  • 目标 MAC / IP（各 6 / 4 字节）：想要通信的目标地址。Request 中目标 MAC 填全 0（因为还不知道）；目标 IP 填写要解析的 IP。Reply 中两个字段都填真实值。

ARP 还有一个变体叫免费 ARP（Gratuitous ARP）：主机上线时主动广播自己的 IP→MAC 映射，让周围所有主机更新 ARP 缓存，也可用于检测 IP 冲突

PPP 协议

PPP 各字段详解

  • 标志 Flag（1 字节，0x7E）：帧定界符，标志帧的开始和结束。若数据中出现 0x7E，用转义 0x7D 0x5E 替代（字节填充）

  • 地址 Address（1 字节，0xFF）：PPP 是点对点链路，两端固定，不需要寻址，此字段固定为广播地址 0xFF（实际无意义）

  • 控制 Control（1 字节，0x03）：固定为 0x03，表示无编号帧（UI 帧），PPP 本身不提供可靠传输（由上层协议负责）

  • 协议 Protocol（2 字节）：指示数据字段承载的上层协议，见表格

  • 数据（0~1500 字节）：上层协议数据，默认 MTU 为 1500 字节（可通过 LCP 协商修改）

  • FCS（2 或 4 字节）：帧校验序列，默认 CRC-16，也可协商为 CRC-32

CSMA/CD 协议

CSMA/CD 不是一个有报文格式的协议，而是以太网的碰撞检测机制，因此并没有报文格式的展示

CSMA/CD 核心概念详解

CSMA/CD 全称为载波侦听多点访问/碰撞检测，三个动作分别是：

  • CS（Carrier Sense，载波侦听）：发送前先检查总线是否有信号，有则等待，总线空闲才可以发送信号

  • MA（Multiple Access，多点访问 / 多点接入）：多台主机共享同一总线，谁都可以发送信号

  • CD（Collision Detection，碰撞检测）：发送的同时持续检测信号，若检测到碰撞（信号叠加失真），立即停止并发出强化干扰信号（Jam），让所有主机都知道发生了碰撞

  • 二进制指数退避（Truncated Binary Exponential Backoff）：碰撞后等待随机时间再重试，第 k 次碰撞后，等待时间从 0∼(2t−1)0 \sim (2^t - 1)0∼(2t−1) 个时槽（51.2 μs）中随机选，其中 t = min(10, k)，最多重试 16 次，超过则放弃

  • 最小帧长 64 字节的意义：确保发送方在帧还未发完时就能收到碰撞信号，从而能检测到碰撞

CSMA/CD 只适用于半双工的有线总线型/共享以太网，现代交换机以太网每个端口独立全双工，不存在碰撞，CSMA/CD 实际上已退出使用，无线局域网改用 CSMA/CA（碰撞避免）

RIP 协议

RIP 各字段详解

  • 命令（8 bit）：1 = Request（向邻居请求路由信息），2 = Response（主动发送或响应请求时发送路由表）

  • 版本（8 bit）：RIPv1=1，RIPv2=2。v1 不支持子网掩码（有类路由），v2 新增掩码字段支持 CIDR

  • AFI（16 bit）：地址族标识，IPv4 = 2。特殊值 0xFFFF 表示该条目是认证信息（明文或 MD5）

  • 路由标记（16 bit）：RIPv2 新增，区分 RIP 学到的路由（内部）和从其他协议重发布进来的路由（外部）

  • 目的 IP / 子网掩码 / 下一跳（各 32 bit）：描述一条路由：去往哪个网络、掩码多长、下一跳是谁

  • 度量值 Metric（32 bit）：跳数，1~15 有效，16 表示不可达（无穷大），这也是 RIP 最大只能支持 15 跳的根本原因

OSPF 协议

OSPF 各字段详解

  • 版本（8 bit）：OSPFv2 固定为 2（IPv4），OSPFv3 为 3（IPv6）

  • 类型（8 bit）：标识五种报文之一（见表格）

  • 路由器 ID（32 bit）：格式与 IP 地址相同，但是逻辑标识符，不是实际用于路由的 IP。通常配置为最稳定的接口地址（如 Loopback 0），在整个 OSPF 域内必须唯一

  • 区域 ID（32 bit）：OSPF 将网络划分为多个 Area，骨干区域固定为 0.0.0.0（Area 0），区域化设计减少了 LSA 泛洪范围，提高了可扩展性

  • 认证（16+64 bit）：支持明文和 MD5 认证，防止伪造 OSPF 报文

OSPF 与 RIP 的最大区别在于：OSPF 每台路由器维护一个完整的拓扑数据库（LSDB），各自运行 Dijkstra 算法独立计算最短路径，而 RIP 只交换距离向量（到各网络的跳数），不知道全局拓扑

BGP 协议

BGP 各字段详解

  • Marker（16 字节）：全置 1，历史上用于认证（现在用 MD5 TCP Option 替代），也用于在 TCP 字节流中同步报文边界

  • Length（16 bit）：整个 BGP 报文（含 Marker）的字节数，最小 19（仅首部），最大 4096

  • Type（8 bit）：见四种报文类型表格

UPDATE 报文中最重要的是路径属性（Path Attributes），这是 BGP 区别于其他路由协议的核心：

  • AS_PATH：记录报文经过的所有 AS 编号，防止路由环路（收到包含自己 AS 号的报文直接丢弃）
  • NEXT_HOP：到达该路由的下一跳 IP
  • LOCAL_PREF：AS 内部优先级，越大越优先（iBGP 传播，不出 AS）
  • MED：多出口鉴别，建议对端从哪个入口进入（越小越优先）

NAT 协议

NAT 也没有具体的报文格式，它是一种运行在路由器 / 防火墙上的地址转换机制，用于把内网的专用地址转换为 NAT 路由器的公网地址

NAT 核心概念详解

NAT 的本质是在 IP 报文通过路由器时改写 IP 地址（和端口），维护一张转换表，使得内网多台主机共用少量公网 IP 对外通信

NAPT（最常见）的转换规则：出向时将<私有 IP:端口>映射到<公网 IP:分配端口>，入向时根据表反向查找还原，整个过程对内网主机和外网服务器透明，在完成转换之后，外部环境只能看到两个路由器的公网 IP 在通信，无法得知内部真正参与通信的内网 IP

DNS 协议

*DNS 各字段详解

  • 事务 ID（16 bit）：客户端随机生成，响应中原样返回，用于匹配哪个响应对应哪个请求

  • Flags（16 bit）：核心控制位，含义见图示：QR 区分请求/响应，RD 请求递归查询，RA 表示服务器支持递归，RCODE 表示查询结果（0=成功，3=NXDOMAIN 域名不存在）

  • 四个计数字段（各 16 bit）：分别指示后面四个区（问题、回答、权威、附加）各有多少条记录

  • Question Section：QNAME 用"标签序列"编码域名（如 www.example.com = 3www7example3com0），QTYPE 指定想查询的记录类型

  • Answer / Authority / Additional Section：都由资源记录（RR）组成，格式相同：NAME + TYPE + CLASS + TTL（缓存时间，秒）+ RDLENGTH + RDATA

  • DNS 解析层次：根服务器（13 组）→ 顶级域（.com/.cn 等）→ 权威服务器，每层服务器只负责自己辖区，通过委派层层指引

HTTP 协议

HTTP 协议报文又具体可以分为请求报文和响应报文，二者的字段存在一定的差异

HTTP 各字段详解

  • 请求行（请求报文的开始行）：由三部分组成 方法 + URL + HTTP/版本号 CRLF，例如 GET /index.html HTTP/1.1\r\n

  • 状态行（响应报文的开始行）：由三部分组成 HTTP/版本号 + 状态码 + 原因短语 CRLF，例如 HTTP/1.1 200 OK\r\n

  • Headers（首部字段）：每行格式 字段名: 值 CRLF，承载大量元信息

    • Host：目标主机名（HTTP/1.1 必须携带，支持虚拟主机）
    • Content-Type：正文的 MIME 类型（如 application/json）
    • Content-Length：正文字节数，接收方据此知道在哪里结束
    • Transfer-Encoding: chunked：分块传输，不需要事先知道总长度
    • Cookie / Set-Cookie：状态管理（HTTP 本身无状态）
    • Cache-Control / ETag / Last-Modified：缓存控制
    • Authorization：身份认证信息

  • 空行：头部和正文之间必须有一个空的 CRLF，这是头部结束的唯一标志

  • Body：请求体（POST/PUT 带数据）和响应体（HTML/JSON/图片等资源内容），格式由 Content-Type 决定，这个字段也统一称为实体主体

汇总

TCP 传输控制协议 传输层 IP 数据报 + 端口号（16 bit） 面向连接、可靠传输；三次握手建连、四次挥手断连；滑动窗口流控；拥塞控制 ① 四次挥手中 TIME_WAIT 在主动关闭方，等待 2MSL ② 序号是字节编号而非报文编号 ③ 确认号 = 期望下一字节，不是已收到的最后字节
UDP 用户数据报协议 传输层 IP 数据报 + 端口号（16 bit） 无连接、不可靠；首部仅 8 字节；适合实时场景（DNS / 视频 / 游戏） ① UDP 校验和在 IPv4 中可选（置 0 = 不用），IPv6 中强制 ② UDP 本身不分片，靠 IP 层分片 ③ 不保证顺序，可能乱序到达
IP（IPv4） 网际协议 网络层 以太网帧 / PPP 帧，协议号字段 尽力而为的无连接数据报转发；IP 地址寻址；分片与重组；TTL 防环 ① IP 是不可靠的，可靠性由 TCP 提供 ② 分片在路由器处进行，重组在目的主机 ③ IP 校验和只覆盖首部，不含数据 ④ TTL 每跳 -1，路由器须重算校验和
ICMP 互联网控制报文协议 网络层 IP 数据报，协议号 = 1 IP 的差错报告与诊断；ping（Echo Request/Reply）；traceroute（TTL 超时）；目的不可达通知 ① ICMP 封装在 IP 内，但属于网络层协议，不是传输层 ② traceroute 原理是令 TTL 从 1 开始递增，逐跳触发 ICMP 超时 ③ ICMP 差错报文不会再触发 ICMP 报文（防止雪崩）
IGMP 互联网组管理协议 网络层 IP 数据报，协议号 = 2，TTL = 1 主机向路由器声明加入 / 离开多播组；路由器周期查询成员；支持 IPTV 等一对多传输 ① TTL 固定为 1，不跨路由器传播 ② 组地址是 D 类 IP（224.x.x.x），不能用作源地址 ③ IGMP 仅管理本地成员关系，多播路由靠 PIM 等协议
ARP 地址解析协议 链路/网络层边界 直接封装在以太网帧，EtherType = 0x0806 将 IP 地址解析为 MAC 地址；广播 Request，单播 Reply；结果缓存在 ARP 表 ① ARP 不经过 IP 层，不能跨路由器（路由器不转发 ARP 广播） ② 免费 ARP 用于检测 IP 冲突 / 更新邻居缓存 ③ ARP 欺骗：伪造 Reply 毒化 ARP 缓存
NAT 网络地址转换 网络层（机制） 路由器维护转换表，修改 IP 头（NAPT 也改端口） 私有 IP ↔ 公网 IP 转换；NAPT 多对一用端口区分连接；解决 IPv4 地址耗尽 ① NAT 破坏端到端透明性，P2P 穿透困难 ② FTP 主动模式需 ALG 特殊处理 ③ NAPT 是多对一，动态 NAT 是一对一但非固定
RIP 路由信息协议 网络层 UDP，端口 520；距离向量算法 域内路由协议；跳数作度量（max 15）；每 30 s 广播完整路由表；水平分割防环 ① 最大跳数 15，16 = 不可达，不适合大型网络 ② 收敛慢，可能出现「计数到无穷」问题 ③ RIPv1 无子网掩码（有类路由），v2 才支持 CIDR
OSPF 开放最短路径优先 网络层 直接封装在 IP，协议号 = 89；链路状态算法 域内路由协议；维护完整拓扑数据库（LSDB）；Dijkstra 算法；Cost = 10⁸/带宽；区域化设计 ① OSPF 直接跑在 IP 上，不用 TCP/UDP ② Router ID 是 32 bit 逻辑标识，不是实际路由用的 IP ③ 骨干区域必须是 Area 0，所有非骨干区必须与 Area 0 相连 ④ DR/BDR 选举在广播网络上进行，减少 LSA 泛洪量
BGP 边界网关协议 网络层 TCP，端口 179；路径向量算法 AS 间路由协议；路径属性（AS_PATH / NEXT_HOP / LOCAL_PREF）做策略路由；eBGP（AS 间）vs iBGP（AS 内） ① BGP 基于 TCP，是应用层报文，但功能属网络层 ② AS_PATH 防环：收到含自己 AS 号的路由直接丢弃 ③ iBGP 水平分割：从 iBGP 学到的路由不再传给其他 iBGP 邻居，需路由反射器解决
以太网 MAC 帧 以太网媒体访问控制帧 链路层 物理介质（双绞线 / 光纤），IEEE 802.3 局域网帧封装；MAC 地址寻址（48 bit）；FCS 差错检测（CRC-32）；最小帧长 64 字节 ① 前导码和 SFD 由物理层处理，不计入帧长 ② 最小帧 64 字节是 CSMA/CD 碰撞检测的需要，不足需填充 ③ EtherType ≥ 1536 是 Ethernet II 帧，≤ 1500 是 802.3 长度字段
PPP 点对点协议 链路层 串行链路（ADSL / 专线），无 MAC 地址 广域网点对点链路封装；LCP 协商链路参数；NCP/IPCP 协商 IP；支持 PAP/CHAP 认证 ① 地址字段固定 0xFF，控制字段固定 0x03，实际无寻址意义 ② 数据中出现 0x7E 需字节填充（转义为 0x7D 0x5E） ③ PPP 本身不提供可靠传输，可靠性靠上层
CSMA/CD 载波侦听多路访问/碰撞检测 链路层（机制） 共享总线 / 集线器以太网，半双工环境 发前侦听、边发边检测碰撞；碰撞后发 Jam 信号、二进制指数退避；最多重试 16 次 ① 现代全双工交换机以太网不存在碰撞，CSMA/CD 已退出使用 ② 第 k 次碰撞从 0 ~ 2^min(k,10)−1 个时槽中随机退避 ③ 无线网用 CSMA/CA（碰撞避免），无法检测碰撞
DNS 域名系统 应用层 UDP / TCP，端口 53；树形域名空间 域名 → IP 地址解析；递归 vs 迭代查询；根 → TLD → 权威三级委派；TTL 控制缓存时间 ① 默认 UDP，超过 512 字节或区域传输才用 TCP ② 递归查询由解析器完成；迭代查询由客户端逐级发问 ③ DNS 的 TTL 是缓存存活时间（秒），与 IP 的 TTL 含义不同 ④ NXDOMAIN（RCODE=3）表示域名不存在，非服务器错误
HTTP / HTTPS 超文本传输协议 应用层 TCP，端口 80（443）；请求-响应模型 Web 资源传输；无状态（靠 Cookie/Session 维持）；HTTP/1.1 持久连接；HTTP/2 多路复用；HTTPS = HTTP + TLS ① HTTP 本身无状态，Cookie 是应用层机制，不是协议本身的状态 ② 301 永久重定向（浏览器缓存新地址），302 临时（每次仍问服务器） ③ HTTP/2 解决应用层队头阻塞，但 TCP 层仍有；HTTP/3 用 QUIC 彻底解决 ④ HTTPS 中 TLS 位于传输层之上、应用层之下

总结：

整个计算机网络的体系可以用一句话概括：不同层次解决不同范围的通信问题，设备工作在哪一层，就能理解哪一层的信息

  • 从小到大的网络范围：主机内部（进程通信，靠端口） → 同一局域网（靠 MAC 地址，交换机转发） → 跨不同网段（靠 IP 地址，路由器转发） → 跨自治系统（靠 BGP，AS 之间路由） → 全球 Internet

  • 设备与层次的对应：中继器/集线器工作在物理层，只懂信号，不懂帧，无法隔离冲突域；网桥/交换机工作在链路层，读 MAC 地址，每个端口是独立冲突域；路由器工作在网络层，读 IP 地址，隔离广播域，连接不同网段

  • 数据流动的核心规律：发送时，数据自上而下逐层封装，每层加上本层的首部（应用数据 → TCP 段 → IP 报文 → 以太网帧 → 比特流）。接收时，自下而上逐层解封装，每层剥掉本层首部，将数据交给上层，在中间路由器处，只进行到网络层，剥掉旧帧头、查路由表、封新帧头，再往下发，路由器不会关心更上层的传输层和应用层

  • 最关键的两个规律：IP 地址（源 / 目的）全程不变，端到端标识通信两端；MAC 地址每跳更换，路由器的每个接口都有自己的 MAC，进来的帧目的 MAC 是路由器，出去的帧源 MAC 换成路由器出口的 MAC，目的 MAC 变为下一跳的 MAC，这需要 ARP 协议实时解析
