传输层

传输层协议概述

进程之间的通信

传输层（Transport layer）又称为运输层，但后者是不严谨的叫法

传输层向它上面的应用层提供通信服务，其功能为：

  • 实现可靠传输：包括差错控制，顺序控制，拥塞控制，连接控制，分组排序与去重

传输层和网络层相比：

  • 网络层实现主机之间的逻辑通信
  • 传输层实现应用进程之间的逻辑通信，传输层是真正的端到端通信，包括复用（multiplexing）和解分（demultiplexing）

传输层的主要协议：

  • TCP 协议：可靠传输协议
  • UDP 协议：不可靠传输协议

考传输层端到端通信：

传输层两个主要协议

传输层两个主要协议如下：

  • 传输控制协议 TCP（Translation Control Protocol）
  • 用户数据报协议 UDP（User Datagram Protocol）

传输的数据单位：

  • OSI 术语把数据传输单位称为 TPDU（Transport Protocol Data Unit）
  • TCP/IP 体系把数据传输单位称为 TCP 报文段（segment）或 UDP 用户数据报

适合使用 UDP 的场景：传送数据之前不需要建立连接，收到 UDP 报文以后也不需要给出确认，不可靠，无连接，时延小

  • 适合较大视频的传输，或较大数据包的下载（其中丢失几个字节无关紧要）

适合使用 TCP 的场景：数据传输之前必须建立连接，数据传输结束以后要释放连接，不提供广播或多播服务，由于 TCP 要提供可靠的面向连接的传输服务，因此不可避免增加了许多开销，包括确认、流量控制、计时器以及连接管理等

  • 适合登录时账号密码的传输，对精确度要求较高，传输时不能出错

传输层的端口

传输层需要为多个应用进程提供服务：复用与分用

  • 复用：应用层多个应用进程通过传输层发送数据
  • 分用：传输层收到的数据必须交付给指明的应用进程

传输层必须提供区分上层应用进程的手段：也就是端口（port）

这里之所以选择端口而不选择进程 ID，主要是考虑了：

  • 进程 ID 的定义依赖于操作系统，不同操作系统定义的进程 ID 格式，长度可能都不同
  • 一个应用进程可能需要和网络中的多个主机中的进程通信，建立多条连接

注意：这里提到的端口为软件端口，不同于计算机中的硬件 I/O 端口和交换机/路由器上的物理端口

TCP/IP 协议使用 16 位整数作为端口号，包括源端口号和目的端口号

端口号有以下三种分类：

  • 熟知（well-known）端口号或系统端口号：端口号数值一般在 0∼10230\sim10230∼1023，例如 http 服务使用 80，FTP 服务使用 21
  • 登记端口号：端口号数值一般在 1024∼491511024\sim491511024∼49151，给没有熟知端口号的应用程序使用，需要在 IANA 登记，以防重复
  • 客户端口号或短暂端口号：端口号数值一般在 49152∼6553549152\sim6553549152∼65535，给客户进程临时使用（比如浏览器临时打开一个网页访问某个网站，会给客户进程临时分配一个端口号，访问完成后就释放掉）

用户数据报协议 UDP

UDP 概述

UDP 只在 IP 数据报服务之上增加了很少一点的功能，即端口和差错检测

  • 区分于 IP 数据报只对包头进行差错检测，UDP 会对整条数据做差错检测

UDP 为不可靠传输协议，但具有自身特点，与 TCP 分别面对不同的应用

UDP 协议的特点：

  • 无连接，即发送数据之前不需要建立连接
  • 使用尽最大努力交付，不保证可靠交付
  • 面向报文使用，没有拥塞控制，很适合多媒体通信的要求
  • 支持一对一，一对多，多对一，多对多的交互通信
  • 首部开销小，只有 8 个字节

UDP 首部格式

UDP 数据报包括两个字段：首部和数据字段

  • 首部有四个字段，8 字节，四个字段分别是：
    • 源端口：源端口号
    • 目的端口：目的端口号
    • 长度：UDP 数据报长度
    • 校验和：UDP 数据报的校验和
  • 伪首部（pseudoheader）：仅在计算校验和时使用，不会实际被传输

IP 数据报的首部 protocol 字段标为 17 时表示数据部分采用 UDP 数据报，UDP 数据报本身也有一个首部，这个首部最前面会包含一个伪首部，用于计算校验和，但数据报传输时就会被去掉

需要注意整数在网络传输过程中的标识，常见的两种字节序：

  • 小端序（Little endian）：低位字节在低地址（低字节在前）
  • 大端序（Big endian）：高位字节在高地址（高字节在前）

网络中通常使用 Big endian，Arm 架构也采用 Big endian，常见 windows 架构采用 Little endian，为了使程序独立于硬件平台，编程时会维护相关转换字节序的宏，比如 htonl(), htons(), ntohl(), ntohs()

校验和的计算：与计算 IP 数据报的包头校验和类似

  • 计算校验和时要添加伪首部，奇数字节要填充 0 字节

UDP 首部格式

在接收方，如果在目的端口号上没有对应的进程接收数据，则会向源主机返回 ICMP 的终点不可达报文

  • 这个性质可能会被某些 scanner 扫描软件所利用，该软件通过向各个端口发送 UDP 数据报，如果没收到对应的 ICMP 回应，则说明该端口有进程在工作，可以根据该进程的特性设计相应的攻击软件进行攻击

传输控制协议 TCP

TCP 主要特点

TCP 是面向连接的传输层协议

  • 数据传输之前必须先建立连接，数据传输完毕以后要释放连接

每一条 TCP 连接只能有两个端点（endpoint），每一条 TCP 连接只能是点对点的（一对一）

TCP 提供的是可靠交互服务

  • 无差错，不丢失，不重复，按序到达（由发送缓存和接收缓存保证）

TCP 提供全双工通信

  • 在一个连接上，通信双方可同时向对方传输数据

面向字节流

  • 在 TCP 连接上传输的应当是字节流，也就是以字节为单位进行传输
  • 应用程序以数据块为单位和 TCP 交互，但 TCP 将其视为无结构的字节流
  • 发送方应用进程发出的数据块与接收方应用程序收到的数据块可能没有一一对应关系，但数据保持一致

面向流的概念：

TCP 通信的主要特点：

  • TCP 连接是一条虚连接（逻辑上的连接）而不是一条真正的物理连接
  • TCP 对应用进程一次把多长的报文发送到 TCP 的缓存中这件事是不关心的，也就是说 TCP 不关心上层应用进程以何种结构 / 方式把待传输的数据写入 TCP 缓存中，反正都是当做字节来看
  • TCP 根据对方给出的窗口值和当前网络拥塞的程度来决定一个报文段应该包含多少字节（采用滑动窗口协议）；而 UDP 发送的报文长度是由应用进程给出的
  • TCP 可以把过长的数据块划分短一些再传送（长→短）
  • TCP 也可以等待积累足够多的字节后再构成报文段发送（短→长）

TCP 协议和 UDP 协议对比：

TCP 协议 UDP 协议
是否建立连接 传输数据前后需要建立和释放连接 无连接
是否可靠 可靠（无差错，不丢失，不重复，按序到达） 不可靠（没有拥塞控制，不保证可靠交付）
通信方向 全双工通信 双向通信（因为没有连接的概念，所以算不上全双工）
传输对象 面向字节流 面向报文段
连接数量 只能是一对一建立连接 支持一对一，一对多，多对一，多对多
连接端点 套接字（IP 地址 + 传输层协议的端口） 传输层协议的端口

TCP 连接

TCP 把连接作为最基本的抽象，每一条 TCP 连接都有两个端点，TCP 连接的端点不是主机，不是主机的 IP 地址，不是应用进程，也不是传输层的协议端口，TCP 连接的端点叫做套接字（socket）或插口

  • 区分 UDP：UDP 连接的端点就是端口

套接字：IP 地址 + 端口号

各个字段长度与含义：

  • 源端口和目的端口字段：各 2 字节

  • 序号字段：4 字节

    • 在 TCP 连接中传送的数据流中的每一个字节都有序号
    • 序号字段指本报文段所发送的数据报中的第一个字节的序号，以字节为单位

  • 确认号字段：4 字节

    • 期望收到对方的下一个报文段的数据的第一个字节的序号，要求小于这个序号的数据都正确被接收
    • 比如收到对方的报文段中的序号为 501，数据长度为 200 字节，则返回报文段确认号为 701，表示下一次希望收到对方发送首字节序号为 701 的数据

  • TCP 连接是全双工，通信双方可相互发送数据，因此应答与数据可以一起发送给对方

  • 数据偏移（也称为首部长度）：4 bit，TCP 报文段的数据起始位置的偏移，同时也是 TCP 报文首部的长度，单位是 32 位字（即一个字为 4 字节）

  • 4 bit 的数据偏移可表示的最大字段长度为 64 字节

  • 保留字段：6 bit，保留，没实际作用

  • 紧急 URG：1 bit，设置为 1 表示紧急指针字段有效，即有紧急数据，需要尽快传送

  • 确认 ACK：1 bit，设置为 1 表示确认号字段有效

  • 推送 PSH：1 bit，设置为 1 表示接收方需要尽快向应用进程交付此报文段，而不是等到整个缓存填满

  • 复位 RST：1 bit，设置为 1 表示 TCP 连接出现严重差错（比如主机崩溃），需要释放连接后重新建立连接

  • 同步 SYN：1 bit，设置为 1 表示请求建立 TCP 连接

  • 终止 FIN：1 bit，设置为 1 表示要求释放 TCP 连接

  • 窗口大小：2 字节，表示自己的接收窗口大小，对方会依据此值设置发送窗口，单位为字节，最大可维护 256 字节的窗口大小

  • 检验和：2 字节，需要同时计算伪首部 + TCP 首部 + 报文数据的校验和

    • TCP 伪首部的格式和 UDP 伪首部格式相同

  • 紧急指针：2 字节，指出本报文段中紧急数据共有多少字节（紧急数据放在所有数据最前面）

  • 选项：长度可变，最长为 40 字节，不同于 IP 数据报包头的可选字段，TCP 协议包头的选项字段常用

    • 最早定义的一种选项：最大报文段长度 MSS（Maximum Segment Size） ，用于告知对方报文段中数据的最大长度，双方可使用不同的 MSS，缺省时 MSS=536 字节
    • 后续增加的选项：窗口扩大选项，时间戳选项，选择确认选项

  • 填充字段：为了确保整个首部长度是 4 字节的整数倍（因为首部长度的单位是 4 字节）

TCP 可靠传输的实现

以字节为单位的滑动窗口

TCP 基于滑动窗口协议实现可靠传输和流量控制，滑动窗口以字节为单位

发送窗口：

  • 在没有收到对方应答的情况下，可以连续把窗口内的数据发送出去，窗口内的序号是允许发送的字节的序号
  • 发送方窗口的大小由数据接收方设置，通过数据报告知发送方
  • 发送方在收到接收方发来的 TCP 报文头部的 acknowledge number 字段，控制发送窗口向前滑动，acknowledge number 字段表示在此序号之前的数据都已被接收方正确接收

发送方和接收方都需要维护一个缓存区：

  • 发送缓存用来存放：（1）发送应用进程传送给 TCP、但尚未发出的数据；（2）已发送，但尚未收到确认的数据
  • 接收缓存用来存放：（1）按序到达的、但尚未被接收应用进程读取的数据；（2）未按序到达的数据

发送方窗口和接收方窗口并不一定是同样大的，一般满足发送方窗口 ≤ 接收方窗口

超时重传时间

TCP 的可靠性传输通过校验和 + 超时重传实现：

  • TCP 每发送一个报文段，就对这个报文段设置一次计时器
  • 如果计时器设置的重传时间已经到达，但发送方还没收到确认，就要重传该报文

超时时间的设置是一个复杂的问题，需要考虑：

  • IP 层提供数据报服务，每个数据报所选择的路由都可能有变化
  • 传输层的往返时间变化较大

（1）首先需要计算加权平均往返时间 RTTSRTT_SRTTS​（RTT，Rount Trip Time），又称为平滑的往返时间

  • 第一次测量得到往返时间 RTTRTTRTT ，RTTSRTT_SRTTS​ 就取为 RTTRTTRTT

TCP 流量控制

TCP 利用滑动窗口实现流量控制，

  • 所谓流量控制（flow control），指的是让发送方的发送速率不要太快，使接收方来得及接收

  • 使用滑动窗口机制可以很方便的在 TCP 连接上实现流量控制

    • TCP 建立连接时，将自己的窗口大小告知对方
    • TCP 数据报的包头都携带有己方的接收窗口大小字段（Window Size），告知对方自己接收窗口的剩余大小（即可接收的字节数）
    • 发送方的窗口应不大于接收方的窗口

接收方的允许接收的窗口大小是可变的，每一次变化都可以通过一次反馈给发送方的 TCP 报文来告知对方：

  • ack：期望收到的字节序列的起始序号
  • Rwnd：允许接收的最大窗口长度

最后如果二者要停止数据传输，则由 B 向 A 发送一条 TCP 报文，将接收方窗口大小设置为 0 并且告知发送方，则发送方就会停止继续发送报文

当 B 想继续接收数据（接收缓存有空间）时，会再次向 A 发送 TCP 报文，告知新窗口的大小，以及期望得到的起始序号，这个报文如果丢失，则二者都会陷入等待，引发死锁

为了解决上述问题，TCP 协议引入了持续计时器（persistence timer）：

  • TCP 为每一个连接都设有一个持续计时器
  • 若发送方收到接收方的零窗口通知，就启动持续计时器
  • 若持续计时设置的时间已到，就发送一个零窗口探测报文段（仅携带 1 字节数据），接收方在确认这个探测报文段时会反馈当前设置的窗口值
    • 如果窗口仍是 0，发送方就重新设置持续计时器
    • 如果窗口不是 0，发送方也收到了接收方新的窗口大小，即可正常发送数据

传输效率控制：

  • 可以用不同的机制控制 TCP 报文段的发送时机

（1）缓存数据达到一定量就发送

  • TCP 维持一个变量，它等于最大报文长度 MSS
  • 只要缓存中存放的数据达到 MSS 字节时，就把这些数据组装成一个 TCP 报文段发送出去

（2）应用进程控制

  • 由发送方的应用进程指明要发送一次报文段，即推送（push）操作

（3）定时发送

  • 发送方的一个计时器到达时间上限，此时就把当前已有的缓存数据装入报文段（但长度不能超过 MSS）发送出去

TCP 拥塞控制

拥塞的概念和成因

拥塞：在某段时间，若对网络中某资源的需求超过了该资源所能提供的可用部分，网络的性能就会变坏，从而产生拥塞，即拥塞产生的条件为

∑对资源的需求>可用资源\sum 对资源的需求>可用资源 ∑对资源的需求>可用资源

  • 网络资源：链路带宽，路由节点缓存及处理能力

拥塞可能的产生原因多种多样，通过简单的增加某种资源的数量往往并不能很好的消除拥塞：

  • 比如：某路由器的缓存容量太小，造成到达该结点的报文丢失，如果增加缓存容量，可能会导致报文在缓存中排队，然如果路由器的处理能力和出口链路带宽没有同步增加，报文排队时间过长，则引起主机超时重发，反而可能增加排队的报文数量，加剧拥塞

拥塞控制的一般原理

网络拥塞的表现：网络性能下降，具体表现为网络吞吐率下降，报文传输时延增大，丢包率增加，用户端响应时间变长

拥塞常常会趋于恶化，这是由 TCP 重传机制引起的

  • 比如：一个路由器由于缓存不足丢弃了部分报文，发送端主机将超时重发，导致网络中备注入了更多报文，从而加剧拥塞

拥塞控制与流量控制的区别：

  • 拥塞控制：防止过多的数据注入到网络中，使网络中的路由器或链路不至过载
  • 拥塞控制是全局性的过程，而流量控制是点对点通信量的控制，是端到端的过程

拥塞控制的作用：

开环控制与闭环控制：

  • 开环控制：在设计网络时实现将有关发生拥塞的因素考虑到，力求网络在工作时不产生拥塞
  • 闭环控制：基于反馈链路，有以下几种措施
    • 监测网络系统以便检测到拥塞在何时、何处发生
    • 将拥塞发生的信息传送到可采取行动的地方
    • 调整网络系统的运行以解决出现的问题

四种常见的拥塞控制方法：

  • 慢启动（slow strat），又称为慢开始
  • 拥塞避免（congestion avoidance）
  • 快重传（fast retransmit）
  • 快恢复（fast recovery）

慢启动与拥塞避免

慢启动基本思路：

  • 开始发送时，如果立即把大量数据注入网络，则可能导致拥塞，因为此时并不知道网络的负荷状况
  • 比较好的办法是，试探性地从小到大逐渐增大发送窗口

拥塞窗口 cwnd（congestion window）：

  • 发送方维持拥塞窗口，它是一个状态变量
  • 拥塞窗口的大小取决于网络的拥塞程度，并且动态变化
  • 发送方让自己的发送窗口等于拥塞窗口，如果接收方反馈的确认信息中，接收窗口 < 拥塞窗口，则发送窗口取较小值，即 min(拥塞，接收)

发送方控制拥塞窗口的原则：

  • 只要没有出现拥塞，拥塞窗口就可以增大，以便把更多分组发送出去
  • 一旦出现拥塞，拥塞窗口就减小，以减少注入到网络中的分组数

判断发生拥塞的依据：

  • 拥塞发生时路由器会丢弃分组
  • 发送端最简单的判断方法：只要没有收到接收端的确认信息，就认为发生了拥塞

慢启动的工作过程：

  • 开始发送报文段时，先设置拥塞窗口 cwnd=1，即设置为一个最大报文段 MSS 的数值（注意，这里的 1 不是 1 字节，而是一个最大报文段的长度）
  • 每收到一个对新的报文段的确认后，将拥塞窗口长度增加一个单位，即增加一个 MSS 的数值（注意这里和下面 cwnd 长度的倍增并不冲突，倍增的过程本质上是拆分成每收到一个确认就 +1 来实现的）
  • 用这样的方法逐步增大发送端的拥塞窗口 cwnd，可以使分组注入到网络的速率更加合理

慢启动工作过程：

传输轮次（transmission round）：

  • 使用慢启动时，每经过一个传输轮次，拥塞窗口 cmd 长度就加倍
  • 一个传输轮次所经历的时间即本次发送所有报文段所经历的总往返时间 RTT~S~，比如：拥塞窗口 cwnd=4，此时的总往返时间 RTT~S~ 即发送方连续发送 4 个报文段，并且收到接收方对这四个报文段的确认，所经历的总时间
  • 因此，传输轮次强调的是：把拥塞窗口 cwnd 所允许发送的报文段都连续发送出去，并且已经收到了对方发送的最后一个字节的确认

为了防止拥塞窗口 cwnd 增长过大导致拥塞，还会设置一个慢启动门限（ssthresh）：

  • cwnd＜ssthresh：使用慢启动算法
  • cwnd＞sshresh：停止使用慢启动算法，而改用拥塞避免算法
  • cwnd=ssresh：使用慢启动算法或拥塞避免算法均可

拥塞避免算法：

  • 让拥塞窗口 cwnd 缓慢增大，即每经过一个往返时间 RTT 就把发送方的拥塞窗口 +1，而不是加倍，使拥塞窗口 cwnd 按线性规律缓慢增长

拥塞发生时的处理：

  • 无论在慢启动阶段还是拥塞避免阶段，只要发送方判断网络出现拥塞（没有按时收到确认），就要把慢启动门限 ssthresh 改成出现拥塞时的发送窗口值的一半（但不能小于 2）
  • 同时把拥塞窗口 cwnd 重新设置为 1，重新执行慢启动算法

这样可以迅速减少注入到网络中的分组数，使得发生拥塞的路由器有足够的时间把队列中积压的分组处理完毕

慢启动 + 拥塞避免的完整流程如下：

  • TCP 连接初始化时，拥塞窗口 cwnd=1，慢启动门限 sthresh=16，设置发送窗口宽度=拥塞窗口宽度
  • 在慢开始阶段，发送端每收到一个确认，就把 cwnd+1，即每经过一个总往返时间 RTT~S~，拥塞窗口 cwnd 就加倍
  • 当拥塞窗口 ≥ 慢启动门限 ssthresh 时，进入拥塞避免阶段，每经过 1 个往返时间 RTT，拥塞窗口 cwnd+1
  • 假定拥塞窗口 cwnd=24 时发生了拥塞，ssthresh 减少为此时 cwnd 值的一半，即 12，cwnd 窗口变回 1，重新开始执行慢启动算法

注意：

  • 慢启动阶段和拥塞避免阶段对往返时间的定义不同，前者是发送所有分组的中往返时间，后者是发送一个分组的往返时间
  • 假设当前处于慢启动阶段，cwnd=8，慢启动门限 ssthresh=12，到下一个传输轮次，cwnd 变为 12 而不是 16，因为一次加倍是拆分成没收到一个确认就 +1 来实现的，只要达到慢启动门限，就从慢启动切换到拥塞避免阶段

慢启动和拥塞避免的两个策略：

  • 乘法减少（multiplicative decrease）：不论在慢启动阶段还是拥塞避免阶段，只要出现一次超时（即网络拥塞），就把慢启动门限 ssthresh 减小到此时对应的拥塞窗口的一半
  • 加法增大（additive increase）：执行拥塞避免算法时，每经过一个往返时间 RTT，把拥塞窗口 cwnd 增加一个 MSS 大小，使拥塞窗口缓慢增大，以防止网络过早出现拥塞

例题1：

发生超时是，新的慢启动门限变成 16/2=8，拥塞窗口 cwnd 变成 1，则再经过 4 个 RTT 时间，前三个时间分别为加倍过程，1→2，2→4，4→8，此时达到新的慢启动门限，第四个时间 cwnd+1，因此最终的 cwnd 值为 9

本例需要有一个前提假设：没有其他限制时（比如接收窗口限制），初始的发送窗口默认等于拥塞窗口大小

因此初始的发送窗口为 4000，发送了两个字段，只有第一个字段被确认，因此发送窗口下沿前移 1000，第二个字段虽然已经发送，但没有收到确认，因此窗口下沿无法继续移动

此时接收窗口长度为 2000，因此发送窗口上沿需要回退 1000，总发送窗口大小为 2000，其中还有 1000 是已经发送还没确认的第二个字段，因此剩余可以发送的字节数只有 1000

快重传和快恢复

快重传（fast retransmit）算法思路：

  • 如果发送方在超时时间内未收到确认报文，说明网络中发生了拥塞，超时时间大约是三倍的 RTT，有可能超出超时时间时，拥塞已经发生了一段时间，我们希望尽可能早的减小拥塞窗口宽度
  • 每当接收方收到一个失序的报文段，它就会发出重复确认（比如接收方没收到 2 号，但是收到了后面的 3,4,5… 号，则收到这些报文段的时候返回的确认信息都是 1 号，说明 2 号没收到）
  • 因此发送方只要连续收到三个重复确认，就应该立即重传对方尚未收到的报文

注意：这里返回确认信息的序号和 TCP 报文段中对确认号的定义不同，可能是 TCP 报文的不同字段（一个连续收到的报文段中的最后一个报文段，一个是期待收到的下一个报文段）

快恢复（fast recovery）算法思路：

  • 当发送端收到连续三个重复的确认时，就进行乘法减小，把慢启动门限 ssthresh 减小为当前拥塞窗口宽度的一半
  • 接下来不执行慢启动算法，而是直接设置拥塞窗口为慢启动门限 ssthresh 减半后的数值，然后开始执行拥塞避免算法（加法增大），使拥塞窗口缓慢地线性增大
  • 这是因为当收到重复确认时，发送方有理由认为现在网络可能没有发生拥塞（仅有拥塞征兆），因为拥塞时不会有连续多个报文段到达接收方

快重传示例：

注意：快重传和快回复的起点也是慢启动，它们只是连续收到三个重复确认时走的一个分支

三个触发条件：

  • 慢启动状态下超过慢启动门限，进入拥塞避免
  • 发生超时，ssthresh 减半，cwnd 变回 1
  • 收到三个重复确认，ssthresh 减半，cwnd 变为减半后的 ssthresh，进入拥塞避免

综合流量控制和拥塞控制来考虑发送窗口的上限值：

  • 流量控制的约束：发送窗口不能超过对方给出的接收窗口 rwnd
  • 拥塞控制的约束：发送窗口不能超过拥塞窗口 cwnd

因此则有：

TCP 连接管理

TCP 传输连接有三个阶段：

  • 连接建立（三次握手）
  • 数据传送
  • 连接释放（两次释放，四次挥手）

连接建立过程中要解决的三个问题：

  • 连接的双方要确知对方的存在
  • 允许双方协商一些参数（如最大报文段长度，最大窗口大小，服务质量等）
  • 能够对传输实体资源进行分配（如缓存，连接表中的项目等）

TCP 连接的建立都是采用客户 / 服务器的方式：

  • 客户（client）：主动发起建立连接的应用进程
  • 服务器（server）：被动等待建立连接的应用进程

TCP 连接建立

TCP 连接的建立采用三次握手（three-way handshake）

  • A 向 B 发出连接请求报文段，其首部中的 SYN=1，并选择序号 seq=x，表明传送数据时的第一个数据字节的序号是 x
  • B 收到连接请求后，如果同意连接，则返回确认报文，其中 SYN=1，ACK=1，确认号 ack=x+1，自己选择的序号 seq=y
  • A 收到后向 B 给出确认，其中 ACK=1，确认号 ack=y+1
  • B 收到确认后，TCP 连接建立，这里的规律是：上一轮传的 seq=t，下一轮回传的 ack=t+1；上一轮的 ack=t+1，下一轮回传的 seq=t+1

之所以采用三次握手而不是两次握手，主要是为了防止失效的连接请求在服务器端占用资源：

  • 客户端发出连接请求，但该数据报在网络中某处滞留，没能及时到达服务端
  • 客户端等待超时后，重新发送链接请求，服务端响应，建立连接
  • 一段时间后，滞留的连接请求又到达服务端，如果仅采用两次握手，此时又会建立一个连接
  • 而每建立一个连接都需要分配相关资源（缓存区，定时器等），相当于长期冗余占用了资源

三次握手也可能带来安全问题，即 TCP SYN Flooding 攻击（该攻击属于 DDos 攻击，即分布式拒绝服务攻击的一种）：

  • 攻击者连续发送大量 SYN 报文，但却不对服务器反馈的 SYN ACK 报文做响应
  • 如上图所示，服务器在收到 SYN 报文并反馈 SYN ACK 报文以后，就会处于 SYN-RCVD 状态，即半开连接状态，占用大量服务器资源
  • 最终的结果会导致服务器内部数据结构堆满，无法响应正常用户的 TCP 连接请求
  • 此类攻击通常会使用 IP 地址伪装，使得对攻击源的定位比较困难

一种应对措施是 Linux 内核所采用的 SYN_Cookies 机制：

  • 服务器返回 SYN + ACK 时，根据自身特有信息（时间戳，IP 地址，端口号等）计算 Cookies，并作为 seq 返回给客户端，在没有收到对方应答时，不会为该连接分配数据结构
  • 相当于把数据结构的分配从第一次握手完成推迟到第三次握手完成

TCP 连接释放

TCP 连接的释放可以由任意一方发起，假定 A 主动关闭连接：

  • A 向 B 发送报文段，其首部的 FIN=1，序号 seq=u；接着 B 发出确认
  • B 发出确认，ACK=1，确认号 ack=u+1，序号 seq=v
  • 此时 A → B 方向的连接已经被释放，连接处于半关闭状态，B 若发送数据，A 仍然要接收
  • 若 B 已经没有要向 A 发送的数据，就向 A 发送连接释放报文（FIN=1，ACK=1）
  • A 收到后，发出确认，其中 ACK=1，确认号 ack=w+1，序号 seq=u+1
  • TCP 连接必须经过时间 2MSL 后才会真正被释放掉（因为要确保该传送的数据都传送完）

关于 MSL：即 Maximum Segment Lifetime，报文段最大存活时间，RFC793 中建议 MSL=2Min，一般使用更小的值

关闭连接前等待 2MSL 时间：

  • 确保 A 发送的最后一个 ACK 报文段可以到达 B
  • 防止旧连接中已经失效的报文段出现在新连接中：当 A 发送完最后一个 ACK 报文段后，再经过 2MSL，就可以使本连接持续的时间内所产生的所有报文段，都从网络中消失，这样可以确保下一个新的连接中不会出现旧连接剩下的报文段
