理论课最后又补充了 IPv6 协议和无线局域网两部分的内容

IPv6

IPv6 概述

IPv4 地址存在的问题：

  • IP 地址空间即将耗尽
  • 安全性问题
  • 服务质量问题

解决 IPv4 地址空间耗尽的策略：

  • 使用无类别编址 CIDR，使 IP 地址的分配更加合理
  • 采用网络地址转换 NAT 结合私有 IP 地址，以节省全球 IP 地址
  • 采用具有更大地址空间的新版本协议 IPv6

对于 IP 家族，即将推出的下一代 IP 协议可以用 IPng 表示（IP Next Generation）

IPv6 地址

IPv6 数据报的目的地址可以是以下三种基本类型的地址之一：

  • 单播（unicast）：传统的点对点通信
  • 组播 / 多播（multicast）：一对多通信
  • 任意播（anycast）：IPv6 增加的类型，任意播的目的站是一组计算机，但在数据报交付时只交付给其中的一个，通常是距离最近的一个

任意播的应用场景：比如一个域名下可能有多台计算机工作，用户访问该域名发送信息时，只需要发送给其中的一个计算机即可，IPv6 即可自动实现这种机制

IPv6 地址为 128 位，采用冒号十六进制记法，每个 16 位的值用一个十六进制表示，各值之间用冒号分隔

  • 例如：686E : 8C64 : FFFF : FFFF : 0000 : 1180 : 960A : FFFF

冒号十六进制有一些约定的简化记法：

  • 零压缩：一连串连续的 0 可以用一对冒号取代（一个地址只能用一次）
  • 可以结合点分十进制的后缀
  • CIDR 的斜线表示法仍然适用

IPv6 首部

IPv6 数据报结构：

  • 首部长度固定为 40 字节，称为基本首部（base header）
  • 取消了不必要的功能，首部字段数减少到 8 个
  • 取消了首部的校验和字段，加快了路由器处理数据报的速度（否则每次转发之前，路由器都要重新计算首部校验和，因为 TTL 变了）
  • 在基本首部的后面允许有零个或多个扩展首部，它们以链表的形式链接起来
  • 所有的扩展首部和数据合起来的部分称为数据报的有效载荷（payload）

IPv6 基本首部的字段组成：

  • 版本（version）：4bit，协议版本，IPv6 的协议版本即为 6
  • 流量类别（traffic class）：8bit，用于区分 IPv6 数据报的类别或优先级
  • 流标号（flow label）：20bit，流指的是从特定源点到特定终点的一系列数据报，流所经过路径上的路由器都需要保证指定的服务质量，所有属于同一个流的数据报都具有相同的流标号
  • 有效载荷长度：16bit，表示数据报除基本首部以外所有字段的长度（以字节为单位），最大值为 64KB
  • 下一个首部（next header）：8bit，相比于 IPv4 的协议字段或可选字段
  • 跳数限制（hop limit）：8bit，数据报发出时设定跳数限制，路由器在转发数据报时将该字段值减 1，当减为 0 时，将该数据报丢弃
  • 源地址：128bit，数据报发送方的 IP 地址
  • 目的地址：128bit，数据报接收方的 IP 地址

扩展首部字段：

原 IPv4 可选首部的功能对应到 IPv6 都放在扩展首部中，数据报途中经过的路由器不会处理扩展首部（除了逐跳选项扩展首部），扩展首部由发送站和接收站的主机处理，路由器处理效率大幅提高

IPv6 共有六种常见的扩展首部：

  • 逐跳选项（路由器唯一处理的扩展首部）
  • 路由选择
  • 分片
  • 鉴别
  • 封装安全与有效载荷
  • 目的站选项

扩展首部和基本首部之间以链表的形式链接，基本首部 / 每一个扩展首部中都有一个下一个首部的指针字段，指向下一个扩展首部 / 数据段

IPv4 向 IPv6 过渡

从 IPv4 向 IPv6 的过渡需要采用逐步演进的方法，需要确保：

  • 新安装的 IPv6 系统向后兼容
  • IPv6 系统能接收和转发 IPv4 的分组，并且能为 IPv4 分组选择路由

目前实现 IPv4 和 IPv6 互操作的主要技术有两种：

  • 双协议栈（dual stack）：主机 / 路由器装有两个协议栈，一个 IPv4 一个 IPv6，根据需要使用不同的协议栈进行通信
  • 隧道技术（tunneling）：将 IPv6 数据报重新封装，再通过 IPv4 网络传输，即 IPv6 数据报作为 IPv4 数据报的数据部分

如果采用隧道技术，重新封装后得到的 IPv4 数据报，包头的协议类型字段为 41

隧道技术

无线局域网 WLAN

WLAN 概述

无线局域网 WLAN 对应的局域网标准为 IEEE 802.11，该网络有两种使用模式：

  • 基础设施模式 / 有架构模式（Infrastructure mode）：每个客户端与一个接入点（AP，Access Point）连接，接入点与其他网络连接，几个接入点可以通过有线网络连接扩展成 802.11 网络
  • 自组织网络模式（ad hoc mode）：多台计算机网络相互连接形成网络，互相发送帧，没有接入点

对于基本服务模式，存在一个基本服务集（BSS，Basic Service Set）：一个基站 + 若干个移动站

  • BSS 内的站之间可以通信，与 BSS 外的通信需要通过基站进行
  • 每个 AP 都会分配一个服务集标识符 SSID 和一个信道，SSID 相当于该基本服务集的名字，我们平时连接 WIFI 连接的就是这个 SSID

关联（association）：移动站与 AP 建立连接，加入该基本服务集

移动站与 AP 建立关联的方法：

  • 被动扫描：移动站等待接收 AP 周期性发出的信标帧（beacon frame）
  • 主动扫描：移动站主动发出探测请求帧（probe request frame），等待 AP 发回的探测响应帧（probe response frame）

重新关联（Reassociation）：移动站改变其首选 AP，加入另一个 BSS

WIFI

WIFI（Wireless fidelity）是无线保真的缩写

Hostpot 即热点，在公共场所提供 WIFI 接入 Internet 服务的地点

802.11 的协议栈：

  • 客户端和 AP 协议相同
  • MAC 子层决定如何分配信道
  • LLC 子层屏蔽 IEEE802 协议之间的差异，同时承上启下

IEEE802.11 的物理层

无线频段：

  • 多数无线频段被政府管理和分配
  • 政府同时预留了无需许可即可免费使用的频段

802.11 的物理层：

  • 红外 IR（早期，已不常用）
  • 2.4GHz 调频扩频 FHSS（早期，已不常用）
  • 直序扩频 DSSS（即码分复用）
  • 正交频分复用 OFDM，即同时改变相位和幅度
  • 多入多出 MIMO

IEEE802.11 的 MAC 层

以太网的 MAC 层使用 CSMA/CD 协议，WLAN 的 MAC 层使用 CSMA/CA 协议，后者不能照搬 CSMA/CD 协议的原因：

  • 无线网络适配器的接收信号强度远小于发送信号的强度，不能像 CSMA/CD 一样进行碰撞检测（边发送边听）
  • 无线信号覆盖范围有限，即使能进行碰撞检测，也可能检测不到碰撞（即隐蔽站问题）

隐藏站问题：

802.11 的 MAC 层提供的两种服务：

  • 分布协调功能 DCF：各站按照 CSMA/CA 协议竞争使用信道
  • 点协调功能 PCF：集中控制方法，由 AP 逐个轮询各站发送数据，避免了碰撞的产生

在实际应用中，DCF 更加常用

帧间间隔（InterFrame Space）：让高优先级帧抢先发送

  • 所有站完成发送后，需要等待一段时间，其余剩下的所有站才可以发送帧
  • 根据要发送的帧类型不同，对应的等待时间间隔也不同
    • SIFS：短帧间间隔，用于 CTS，ACK 等帧，长度 28μs
    • DIFS：DFCF 帧时间间隔，用于 DCF 模式下数据帧，长度 128μs
  • 高优先级帧等待时间较短，可以抢先发送
  • 低优先级帧等待时间较长，等待完毕后如信道忙，则重新等待

802.11 的载波监听机制：

  • 物理载波监听：简单地检查信道，看是否存在有效的信号
  • 虚拟载波监听：802.11 特有的载波监听

虚拟载波监听（Virtual Carrier Sense）：

  • 发送站把它要占用信道的时间放在发送帧中，其他站在收到该信息后，将不在这一段时间内发送数据，大幅减少冲突概率
  • 对应 MAC 帧中的持续时间 Duration 字段（表示本帧结束后还要占用信道多少时间，单位 μs，包括确认帧所需要的时间）
  • 虚拟载波监听的含义在于，其他站并没有监听信道，而是在收到发送站的通知后才不再发送数据

网络分配向量 NAV（Network Allocation Vector）

  • 当一个站检测到正在信道中传送的 MAC 帧首部的持续时间字段时，就调整自己的网络分配向量
  • 网络分配向量指出了需要经过多少时间才能完成本次数据帧的传输，从而使信道转入到空闲状态

发送前退避机制：

  • 站点要发送数据帧，不仅要等待 DIFS 帧间间隔，当发生冲突时，需要退避一段时间后再重新检查信道，信道空闲时才能发送
  • 目的：因为如果大家在检测到信道空闲的瞬间，同时发送信息，则大概率会发生碰撞，因此考虑使用退避机制没让每个站点随机等待不同的时间，从而错开发送时机

CSMA/CA 的退避算法：

CSMA/CA 站点发送数据帧的流程：

  • 第一次发送，且信道空闲：等待 DIFS 后直接发送。

  • 其他所有情况（信道忙、重传、发送第二帧），进入退避流程：随机选取退避时间，开始倒计时；信道忙则冻结计时器，信道重新空闲后等待 DIFS 再继续倒计时；计时器归零后发送

  • 发送后等待 ACK：

    • 收到 ACK：发送成功，若还有下一帧，重新进入退避流程
    • 未收到 ACK：发送失败，CW 翻倍后重新进入退避流程，直到成功

为了减少隐蔽站带来的碰撞，发送站可以对信道进行预约：

  • RTS：Ready to Send，发送站发送的请求发送帧
  • CTS：Clear to Send，目的站返回的允许发送帧

RTS 和 CTS 帧长度很短，虽然增加了开销，但与直接发送较长数据帧导致碰撞相比，开销较小

记住这个图就行：从开始等待 DIFS 到最后发送完 ACK 可以看成一个周期，一共往返了 4 帧，等待了 4 段时间，其中第一段是 DIFS（最长），后面三段都是 SIFS

CSMA/CA 特性总结：

  • 发送前退避
    • 信道从忙变空闲时，所有等待的站点都会同时检测到，如果大家同时发送必然碰撞，退避机制让每个站点在 0~CW 范围内随机选取不同的等待时间，从而错开发送时机，避免碰撞
  • 发送站 / 接收站之间 RTS/CTS 握手（以上面的 A,B,C 模型为例）
    • 发送站 A 先发送一个短帧 RTS（Request To Send）给 B（B 是中间站），B 收到后广播 CTS（Clear To Send），A 和 C 都能听到 CTS，得知信道即将被占用，C 就会等待而不发送，这样隐藏站 C 即使听不到 A，也能通过 CTS 得知信道状态
  • 虚拟载波监听配合 NAV 使用
    • RTS 和 CTS 帧中携带本次传输所需的时间信息，其他站点读取后设置一个 NAV（Network Allocation Vector，网络分配向量） 计时器，在计时器归零前直接认定信道忙，无需真正监听信道，这就是"虚拟"载波监听的含义
  • ACK 确认机制
    • 无线信道容易受干扰，发送方无法像有线网络那样自己检测冲突，所以需要接收方明确回复 ACK 来告知是否成功收到，收不到 ACK 就视为发送失败，触发重传
  • 不同类型设置不同的帧间间隔
    • 高优先级的帧能更快发送出去，ACK 等响应帧只需等待最短的 SIFS，普通数据帧要等更长的 DIFS，所以 ACK 总能在下一个数据帧之前优先发出，保证确认机制正常运作

IEEE802.11 的帧结构

802.11 的帧分为三种类型：数据帧，控制帧，管理帧，下面展示的是数据帧的结构

帧控制字段（Frame Control），一共包含 11 个子段

  • Version：协议版本
  • Type 和 SubType：帧类型（数据，控制，管理）和子类型（RTS，CTS 等）
  • To DS 和 From DS：该帧是发送到 AP 还是来自 AP 连接的网络
  • More Fragment：分片传输时使用
  • Retry：是否是重传帧
  • Power management：指明发送方进入节能模式
  • More Data：发送方还有更多帧需要发送
  • Protected Frame：该帧数据部分是否被加密
  • Order：告诉接收方高层是否按顺序处理帧序列

持续时间字段（Duration）：本帧以及它的 ACK 帧将占用信道的时间（单位 μs）

地址字段（address）：连续三个字段，三个 MAC 地址，国内的两个站之间通信需要经过 AP，由此涉及到三个地址

帧序号（Sequence）：用于重复帧的检测

数据（Data）：帧的有效载荷，最多携带 2312 字节，但通常不会超过 1500 字节

帧校验（Frame check Sequence）：32bit 的 CRC 校验

管理帧和数据帧格式相同，控制帧较短，只有一个地址，没有数据

IEEE802.11 的安全性

两种认证方式：

  • 有认证服务器：配置认证服务器，存有用户名和口令数据库，适用于企业
  • 无认证服务器：移动站和 AP 之间通过预共享秘钥进行认证，适用于家庭和小型应用场合

认证握手过程：4 次握手

  • AP 发出一个随机数（仅使用一次的临时值）用于识别
  • 客户端选取自己的临时值，用自己的临时值，MAC 地址，AP 的 MAC 地址，主秘钥作为参数计算会话秘钥 K~S~，客户端将自己的临时值和消息完整性检查值（MIC）发送给 AP
  • AP 分发一个组秘钥 K~G~，用于后续的广播和组播
  • 客户端确认组秘钥
