Git 操作

通过阅读本文，您可以大致了解用 Git 进行版本控制

版本控制

版本控制：版本控制是一种记录文件内容变化，以便将来查阅特定版本修订情况的系统

  • 版本控制最重要的是可以记录文件修改历史记录，从而仍能够用户能够查看历史版本，方便版本切换

版本控制软件的优势：

  • 方便保存和管理文件
  • 提供客户端工具进行访问
  • 提供不同版本文件的对比功能
  • 支持多人协作开发

Git 基本命令1

命令总览

Git 常用命令如下：

命令名称 作用
git config --global user.name 用户名 设置用户签名
git config --global user.email 邮箱 设置用户邮箱
git init 初始化本地库
git status 查看本地库状态
git add 文件名 添加到暂存区
git commit -m “提交日志说明” 文件名 提交到本地库
git reflog 查看历史记录
git reset –hard 版本号 版本穿梭

设置签名

（1）设置全局用户名和邮箱

说明：

  • Git 在首次安装时必须设置一个用户签名，否则无法提交代码
  • 签名的作用是用于区分不同操作者的身份
  • 用户的签名信息在每一个版本的提交信息中能够看到，以此确定本次提交是谁做的
  • 签名设置完成以后，会在C:\Users\当前用户目录下生成.gitconfig，其中包含设置好的name和email字段，执行cat ~/.gitconfig命令即可查看该文件中的内容

（2）设置局部用户名和邮箱

说明：

  • 去掉命令行中的--global参数，即为设置局部签名
  • 局部签名会在本地的.git目录下保存

初始化本地仓库

Git 版本管理下的分区：

  • 工作区：正在编译的文件，包含已跟踪文件的修改和未跟踪的新文件，修改不会自动保存到 Git
  • 缓冲区：git add后的文件，临时存放准备提交的更改，是工作区和本地仓库之间的缓冲层，存在于.git目录下的index文件
  • 提交区 / 本地仓库：git commit后的文件，即本地历史记录，保存在.git隐藏文件中，存储项目的完整版本历史
  • 远程区 / 远程仓库：git push后的文件，即团队共享的代码，位于远程服务器上（如 GitHub，GitLab 等）

文件的三种状态：

  • 红色：表示文件在工作区，尚未添加到暂存区
  • 绿色：表示当前文件已经参加到暂存区，但是尚未添加到本地库
  • 暂存区空：表示所有文件都已添加到本地库

Git 分支操作

一般正式上线 / 投入使用的系统对应主分支的代码，对应还会有开发分支、测试分支、bug 修复分支，当我们想做增量开发时，为防止影响主分支的功能，往往不会直接动主分支的代码，而是新建一个开发分支，完成开发并测试没有问题以后，再合并回主分支

分支的概念

几乎所有的版本控制系统都以某种形式支持分支，使用分支则意味着可以把当前的工作从开发主线上分离开，避免影响到开发主线

Git 在创建仓库的时候。Git init 默认会自带一个分支，就是所谓的主分支 master，它并不是一个特殊分支，和其他分支没有区别

在版本控制过程中，同时推进多个任务，为每个任务创建一个单独分支，每个程序员在自己的分支中完成代码，开发完成后提交，由管理者负责合并分支到主干 master、测试、发布新版本

使用分支的好处：

  • 同时并行推进多个功能开发，提高开发效率
  • 各个分支在开发过程中，如果某一个分支开发失败，不会对其他分支有任何影响，失败的分支删除重新开始即可

命令名称 作用
git branch -v 查看分支
git branch <分支名> 创建分支
git checkout <分支名> 切换分支
git merge <分支名> 把指定的分支合并到当前分支上

分支合并

基本语法：

在当前分支下，执行git merge，把其他分支的代码合并进当前分支，我们常用的操作即在主分支下，把其他分支的代码合并过来

如果对于每一个文件，主分支和待合并的分支只有一个分支修改了这个文件，则合并时不会产生冲突，而如果两个分支都对同一个文件做了不同修改，则此时会产生合并冲突，需要手动解决冲突

git merge 的性质：

  • 在 A 分支执行 git merge B，如果 B 分支是 A 分支的一个历史版本，即 A 分支在 B 分支的上游，则此时不会执行合并，会直接显示 Already up to dated
  • 分支合并的底层思路是把 B 中的修改更新到 A 中，如果 B 分支内部相对于 A 新增或删除了某文件，则对应文件在合并后也会发生相应的变化（新增或消失）

冲突解决

当产生合并冲突时，当前所在的分支状态会带有 MERGING 标记，比如 master|MERGING，此时产生冲突的文件在执行git status时会变回红色，即又变成未追踪文件

具体来讲，我们需要手动编辑有冲突的文件，删除特殊符号，决定保留要使用的内容，特殊符号指的是：<<<<<<< HEAD 当前分支的代码 ======= 合并过来的代码 >>>>>>> hot-fix

编辑冲突的文件，把>>>>>>，<<<<<<，=======等删除，具体内容保留为正确的状态，再次执行git add和git commit进行提交，即可完成本次合并冲突的解决，提交后当前分支后面的 MERGING 标记消失，变回正常

注意：解决冲突后提交git commit时不能带有具体的文件名，因为我们解决的冲突可能存在于多个文件，因此这里不需要指定具体的文件名

Git 代码托管服务

我们可以借助互联网上提供的一些代码托管服务来实现远程仓库，其中比较常用的有 GitHub、码云、GitLab 等

  • gitHub（ 地址：https://github.com/ ）

是一个面向开源及私有软件项目的托管平台，因为只支持 Git 作为唯一的版本库格式进行托管，故名 gitHub

  • 码云（地址： https://gitee.com/ ）

是国内的一个代码托管平台，由于服务器在国内，所以相比于 GitHub，码云速度会更快

  • GitLab（地址： https://about.gitlab.com/ ）

是一个用于仓库管理系统的开源项目，使用 Git 作为代码管理工具，并在此基础上搭建起来的 web 服务

Git 版本回退

在 Git 中有 git reset 和 git revert 两种版本回退操作，它们各自对应的使用场景如下：

（1）已提交，没有 push 时：

  • git reset --soft <版本号>：撤销对应版本号的 commit
  • git reset --mixed <版本号>：撤销对应版本号的 commit 和 add

（2）已提交，且已经 push 时：

  • git revert <版本号>：会在对应版本号对应提交的基础上，创建一次新的提交 commit‘，对应内容即为撤销对应版本号提交的内容，会保留原本被撤销的那次提交
  • git reset --hard <版本号>：撤销并舍弃当前版本号及其之后所有的提交记录，相当于毁尸灭迹

（1）git reset 示意图

适用场景：如果一不小心把一个子分支推送到主分支，对主分支造成了破坏，此时可以考虑 git reset --hard 清除掉这次子分支提交，确保主分支代码正确

（2）git revert

适用场景：如果只是单纯想恢复一次错误的提交（最常见的是撤销最近一次的提交），则最好使用 git revert，如果只是想撤销最近的一次提交，使用 git revert HEAD 即可

版本穿梭对应的版本号可以通过 git reflog 或者 git log 检出
