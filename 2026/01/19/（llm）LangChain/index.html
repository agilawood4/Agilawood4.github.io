LangChain

通过阅读本文，您可以了解llm中常见的langchain框架的相关知识和常用模块，对大语言模型的调用有一个初步的认识

LangChain 概述

大模型相关介绍

大模型（Large Model）是指具有大量参数和复杂结构的机器学习模型，能够处理海量数据，完成各种复杂的任务，如自然语言处理，计算机视觉，语音识别等

大语言模型（Large Language Model）通常是具有大规模参数和计算能力的自然语言处理模型，如 OpenAI 的 GPT-5 模型，这些模型通过大量的数据和参数进行训练，以生成人类类似的文本或回答自然语言的问题

大模型相关岗位：

其中应用与开发方向是大模型最值得关注的方向，LangChain框架正是用于帮助我们对大模型进行高效的开发和应用

现有大模型存在的普遍问题：

  • 数据更新存在截止时间，无法实时获取最新的数据
  • 无法连接互联网（即联网搜索功能）
  • 无法查询数据库，或接入私有数据
  • 无法对接外部工具，或调用第三方 API

LangChain 介绍

LangChain 中的 Lang 指的是 Language，即大语言模型，而 Chain 为链的含义，即将大语言模型与外部数据以及各种组件连接成链，以此构建 AI 应用程序

LangChain 提供了一些现成的链式组装，用于完成特定的高级任务，让复杂的逻辑变得结构化，易组合，易扩展

LangChain 的功能

LangChain 作为辅助构建 LLM 应用的全套工具集，提供了以下六个核心功能：

  • prompt 构建
  • LLM 接入
  • 记忆管理
  • 工具调用
  • Agent 开发
  • RAG

LangChain 的包

LangChain 中常见的包，其中我们最常使用的就是from langchain import ...和from langchain_core import ...这两个包：

  • langchain：包含构建使用llm的应用所需的所有实现的主入口点
  • langchain_core：LangChain 生态系统中的核心接口和抽象
  • langchain_text_splitters：用于文档处理的文本分割工具
  • langchain_mcp_adapters：在 LangChain 和 LangGraph 应用中提供 MCP 工具
  • langchain_tests：用于验证 LangChain 集成包实现的标准化测试条件
  • langchain_classic：遗留的 angchain 实现和组件

LangChain 核心组件

LangChain 的核心组件如下：

  • Model IO：标准化大模型的输入和输出，包含提示词模版，模型调用，格式化输出
  • Chains：将多个组件组合成一个完整的流程，方便链式调用
  • Retrieval：对应 RAG，检索外部数据，作为参考信息输入 LLM 辅助生成答案
  • Agents：自主选择工具并规划执行步骤

Model I/O 模块

功能：标准化大模型的输入和输出，包含提示模版，模型调用和格式化输出

  • format（输入格式化）：通过模版管理大模型的输入，将原始数据格式化成模型可以处理的形式，插入到一个模版中，然后送入模型进行处理
  • predict（模型预测）：调用llm接收输入，进行预测或生成回答
  • parse（输出解析）：规范化模型的输出，比如将模型输出格式化为json

Retrieval 模块

功能：对应 RAG，检索外部数据，作为参考信息输入大模型辅助生成答案

  • Source：多种类型的数据源，包括视频、图片、文本、代码等
  • Load：将多源异构数据统一加载为文档对象
  • Transform：对文档进行转换和处理，比如将文本切分为小块
  • Embed：将文本编码为向量
  • Store：将向量化后的数据存储起来
  • Retrive：从文本库中检索相关的文本段落

Agents 模块

功能：搭建 Agent，自主规划执行步骤并调用工具完成相关任务

大模型的开发与应用

大模型开发场景

（1）RAG 开发

目前的大模型存在两个问题：

  • 大模型知识冻结：随着llm规模扩大，训练成本与周期相应增加，模型无法实时学习到最新的信息或动态变化，导致llm难以对“今天天气怎么样”，“为我推荐近期热门电影”这类时间敏感的问题做出回答
  • 大模型幻觉：涉及到大模型从未在训练过程中学习过的信息时，大模型无法给出准确的答复，转而开始臆想和编造答案

为了防止大模型因为不知道答案而开始胡编乱造，我们可以为大模型外挂一个知识库，让大模型能够在这个额外提供的知识库中得到提示和参考，从而提高大模型回答的准确性，这就是 RAG 技术

（2）Agent 开发

充分利用llm的推理与决策能力，通过增加规划，记忆和工具调用的能力，构造一个能够独立思考，逐步完成给定目标的 Agent

现代 Agent 架构：

  • LLM：提供推理、规划和知识理解能力，是 AI Agent 的决策中枢
  • Memory：确保 Agent 像人类一样，能够保存其学到的知识以及交互习惯，使得其在处理重复工作时调用以前的经验，避免用户进行定量重复交互
    • 短期记忆：存储单次对话周期的上下文信息，属于临时信息存储机制，受限于模型的上下文窗口长度
    • 长期记忆：可以横跨多个任务或时间周期，可存储并调用核心知识，非即时任务，长期记忆可以通过参数微调（固化知识），知识图谱（结构化语义网络），**向量数据库（相似性检索）**等方式实现
  • Tools：调用外部工具（如 API，数据库等）扩展能力边界
  • Planning：通过任务分解，反思与自省框架实现对复杂任务的处理，如通过思维链（Chain of Thought）将目标拆解为子任务，并通过反馈优化策略
  • Action：实际执行决策的模块，涵盖软件接口操作和物理交互

大模型开发方向

（1）纯Prompt形式

以提示词作为操作大模型的唯一接口，应用程序给大模型提供prompt，大模型反馈response给应用程序

（2）Agent + Function Calling形式

由 Agent 主动提出要求，需要对接外部系统时，AI 要求执行某个函数

  • 把 AI 当人看：你问 ta「我明天去杭州出差，要带伞吗？」，ta 让你先看天气预报，你看了告诉ta，ta 再告诉你要不要带伞

（3）RAG形式

当需要额外补充某一领域的相关知识时可以考虑使用RAG，比如通过**Embedding把文字转化成向量，进而存入向量数据库**，通过向量搜索，根据输入向量，找到最相似的向量

  • 开卷考试答题时，到书上找相关内容，再结合题目组成答案

（4）Fine-tuning微调

成本最高，效果一般也最好，在前面的方法训练出来的大模型都解决不了问题时，可以通过模型微调 / 精调提高模型的性能

  • 闭卷考试答题时，先努力学习相关知识，长期记住，最终活学活用

模型初始化相关参数

通过调用init_chat_model方法用来初始化一个模型，该方法参数列表如下：

  • model：模型名称或标识符
  • model_provider：模型提供商，gpt系列提供商为openai，deepseek系列提供商直接写deepseek即可

  • temperature：控制模型输出的随机性，数字越高，回答越有创意；数字越低，回答越确定
  • timeout：在取消请求之间，等待模型响应的最大时间，以秒为单位
  • max_tokens：限制响应中的总 tokens 数量，控制输出长度
  • max_retries：请求失败时系统尝试重新发送请求的最大次数
Token 概念可视化

  • 一个中文 token 大约为 1-1.8 个汉字
  • 一个英文 token 大约为 3-4 个字母

对话式与非对话式模型

对话式模型：可以和用户进行交互对话（OpenAI），即对用户发出内容进行回复

非对话式模型：对用户提供的内容进行续写（ChatOpenAI）而非回复（比如用户询问“你好”，大模型可能会帮用户扩写一些问候所用的常见话题或询问，“你好”的含义等，但不会回复“我很好”这样的回答）

非对话式模型的使用场景：

  • 一次性交互的任务，比如翻译某句话

对话模型的 Message

对话模型的输入可以是文本提示，消息提示或字典格式

（1）直接在model.invoke()中传入文本，适用于不需要保留对话历史的直接生成任务

（2）使用消息列表，把消息对象列表输入模型，方便管理对话历史，包含系统指令以及处理处理多模态数据

常见消息类型：

  • SystemMessage：代表一组初始指令，用于引导模型的行为，可是使用系统消息来设定语气，或定义模型扮演的角色
  • HumanMessage：表示用户输入
  • AIMessage：模型生成的响应，包括文本内容，工具调用或元数据
  • ToolMessage：表示工具调用的输出

消息类型（3）使用消息列表，通过字典方式构建消息列表，按照OpenAI调用格式创建字典列表组成消息，一条消息至少需要包含role和content两对键值对

其中role的值被限定在human, user, ai, assistant, function, tool, system, developer几类中，不支持除此之外的其他角色设置

模型调用方法

除了最常见的invoke()方法之外，langchain模型还提供了其余可供选择的调用方法：

  • invoke/ainvoke：将单个输入转换为输出
  • batch/abatch：批量将多个输入转换为输出
  • stream/astream：从单个输入生成流式输出

其中带有a前缀的方法表示是异步调用大模型，需要与asyncio和await语法一起使用以实现并发

流式 / 非流式输出

（1）非流式输出：LangChain 与 LLM 交互时的默认行为，是最简单、最稳定的语言模型调用方式，当用户发出请求后，系统在后台等待模型生成完整响应，然后一次性把全部结果返回

（2）流式输出：流式输出是一种更具有交互感的输出方式，用户不再需要等待完整的答案，而是能够看到模型逐个 token 地实时返回内容，通过stream()方法实现

批量调用

批量调用指的是把一组独立的请求一次性发给模型，进行并行处理，通过batch()方法实现

注意：batch默认没有依赖底层 API 的原生批量接口，而是使线程池并行地执行多个invoke()，因此其对于 I/O 密集型任务很有效

如果底层 API 有提供批量接口（一次请求中有多条prompt），子类也可以重写batch方法来使用批量接口，这样效率会更高

同步 / 异步调用

（1）同步调用：每个操作依次执行，直到当前操作完成后才开始下一个操作，总执行时间是各个操作执行时间的总和

（2）异步调用：允许程序在等待某些操作完成时继续执行其他任务，而不是阻塞等待，这在处理 I/O 操作（如网络请求、文件读写等）时特别有用，可以显著提高程序的效率和响应性

Prompt Template

在应用与开发中，固定的提示词限制了模型的灵活性和适用范围，Prompt Template即为一个模板化的字符串，我们可以将变量插入到模版中，从而创建出不同的提示

提示词模版以字典违输入，其中每个键都代表要填充的提示词模版中的变量，输出一个PromptValue，这个PromptValue可以传递给聊天模型，也可以转换为字符串或消息列表，其存在的目的即便于在字符串和消息之间转换

常见的提示词模版有PromptTemplate（字符串提示模版）和ChatPromptTemplate（聊天提示模版）

Output Parsers

OutputParsers即输出解析器，在应用开发中，大模型的输出可能是下一步逻辑处理的关键输入，因此在这种情况下，规范化输出是必须要做的任务，以确保应用能够顺利进行后续的逻辑处理

语言模型返回的内容通常都是文本字符串，而实际 AI 应用开发过程中更希望模型可以返回更直观，更格式化的内容，LangChain提供了输出解析器（OutputParser）将模型输出解析为结构化数据

常用的输出解析器有：

  • StrOutputParser：把模型输出解析为纯文本（str 类型）
  • JsonOutputParser：把模型输出解析为 json 格式
  • Structured OutputParser：大模型自带的 json 结构化输出接口

Chains 模块

Runnable 与 LCEL

Runnable

Runnable是LangChain中可以调用、批处理、流式传输，转换和组合的工作单元，通俗来讲，我们前面所学到的输出解析器（如OutputParser），提示词模版（如PromptTemplate），大语言模型（如ChatOpenAI）在底层都继承自Runnable类，该类为这些上层组件提供了一些方便调用的统一接口

  • invoke / ainvoke：将单个输入转换为输出
  • batch / abatch：批量将多个输入转换为输出
  • stream / astream：从单个输入生成流式输出

以invoke()方法为例，假设没有这个统一的调用方式，则每个组件都使用各自的调用方式，组合时需要手动适配：

  • propmt_template.format()：提示词渲染
  • openai_llm.generate()：模型调用
  • output_parser.parse()：解析输出

LCEL 管道

基于Runnable对一些常见接口的统一实现，我们可以通过管道符|把各个组件进行组合，系统会自动处理好类型匹配和中间结果传递，快速实现提示词→模型调用→输出解析的管道式调用

最后再统一调用chain的invoke()方法，即可完成整个调用流程

LCEL（LangChain Expression Language）指的是LangChain表达式语言，即从现有的Runnable构建新的Runnable的声明式方法，用于声明、组合和执行各种组件（包括模型，提示词，工具，函数等）

我们称使用 LCEL 创建的Runnable称为链，链本身也是Runnable

LCEL 两个主要的组合原语是RunnableSequence和RunnableParallel，许多其他组合原语可以被认为是这两个原语的变体

RunnableSequence 可运行序列

RunnableSequence按顺序链接多个可运行对象（要求每个对象必须是Runnable类型），其中一个对象的输出作为下一个对象的输入

LCEL 重载了|运算符，以便使用两个Runnable对象创建RunnableSequence对象

RunnableParallel 可并行运行

RunnableParallel同时运行多个可运行对象，并为每个对象提供相同的输入

一个比较常用的场景即实用RunnableParallel同时调用多个大模型，询问其相同的问题，横向对比不同大模型对同一个问题回答的质量

另一个比较常见的场景是构建多套提示词模版，同时调用一个大模型，让大模型一次性回答多个不同的问题

这里实例化RunnableParallel的传参方式有两种，其本质都是指一个字典：

  • {"key1":runnable1, "key2":runnable2, ...}：按照字典的格式，要求值必须是Runnable对象，键是给这个值所起的别名
  • key1=runnable1, key2=runnable2, ...：按照赋值的格式，要求右值必须是Runnable对象，左值是给右值所起的别名

对应的RunnableParallel实例化对象在调用invoke()方法时，该方法接受一个dict（如果只有一个键值对，可以省略键，只传入值），其执行的本质是把这个字典分配到实例化对象中每一个可运行对象上，让每个runnable对象都接受这个dict作为入参执行

RunnableLambda 可运行λ\lambdaλ

RunnableLambda将 Python 可调用函数转换为Runnable对象，使得函数可以在同步或异步上下文中使用，通俗来讲，即可以在其中传入一个 λ\lambdaλ 函数作为Runnable对象

RunnablePassthrough 可运行透传

RunnablePassthrough接收输入并将其原样输出，属于LangChain的 LCEL 体系中的无操作节点，用于在流水线中透传输入或保留上下文，也可以用于向输入中添加键

RunnableBranch 可运行分支

RunnbaleBranch使用(条件, Runnable)这样的元组队列表和默认分支进行初始化，对输入进行操作时，选择第一个计算结果为 True 的条件，并在输入上运行该条件对应的Runnable对象，如果没有任何条件为 True，则在输入上运行默认分支

RunnableWithFallBacks 可运行带回退

RunnableWithFallbacks 使得 Runnable 失败后可以回退到其他 Runnable，可以直接在 Runnable 上使用 with_fallbacks 方法

这个实例同时给出了一个chain构造过程中的易错点，链中的每个对象都必须是Runnable类型的对象，而提示词模版PromptTemplate本身是可运行对象，但是执行format()方法传入具体变量值以后则变成字符串类型，此时不能作为chain中的一个组成对象

因此我们在建链的时候只能先写入提示词模版，在调用chain.invoke()方法的时候再传入对应的{"key":"val",...}填充具体的提示词内容即可
